## 一、执行摘要

### 1.1 核心结论

**XG API数据源 ❌ 不满足毕业生离校申请审批系统完整需求。**

**符合度：**
- 基础字段覆盖：27% (3/11)
- 必填字段覆盖：57% (4/7)
- **关键业务字段覆盖：0% (0/3)** ← 核心阻断
- 数据质量评分：95/100 (A级)

**关键发现：**
- ✓ 提供的字段数据质量优秀
- ✗ 缺失3个P0关键业务字段
- ✗ 无法独立支持核心业务流程
- ✗ 只能作为补充信息源，不能替代CSV

### 1.2 影响评估

| 影响等级 | 问题数 | 说明 |
|---------|-------|------|

## 三、不满足条件详细分析

### 3.1 P0级别：关键业务字段缺失（阻断核心流程）

#### P0-1: class_id（班级ID）缺失

**不满足表现：**
- XG API实际采样20条记录，均无class_id或等价字段
- 无法从department、parent_dep、或其他关联对象推导唯一班级标识
- 32,039条总记录中，无任何记录包含可用班级编码

**业务影响：**
1. **审批人分配失败**
   - 系统依赖 `Student.class_id → ClassMapping.class_id → ClassMapping.counselor_id → Counselor`
   - 缺失class_id导致审批链路首个分派点无法落地
   - 学生提交申请后无法自动进入辅导员审批环节

2. **班级映射表无法建立**
   - ClassMapping表以class_id为主键
   - 无法维护"一班一辅导员"对应关系
   - 整个辅导员分配机制失效

3. **新用户创建阻断**
   - User模型要求学生必填class_id
   - API缺失此字段导致无法创建完整用户记录
   - 必须由CSV补充或拒绝创建

**根本原因：**
- XG用户API设计未包含学生班级归属信息
- 可能需要独立的教务/学工班级管理API

**缓解措施：**
- ✓ 已实施：CSV导入维护class_id
- ✓ 已实施：API同步时不覆盖class_id字段
- ⚠️ 限制：新用户必须由CSV创建

---

#### P0-2: is_graduating（毕业生标识）缺失

**不满足表现：**
- XG API返回32,039条用户记录
- 实际需求：当届毕业生约1,000人
- 无任何字段标识"是否为本批次毕业生"
- 无法区分在校生、往届生、当届毕业生

**业务影响：**
1. **申请资格判定失败**
   - 系统要求 `is_graduating=true` 才能提交离校申请
   - API缺失标识导致无法自动判定资格
   - 可能错误开放申请入口给非毕业生

2. **批次管理混乱**
   - 无法按毕业批次筛选用户
   - 无法自动关闭/开放特定批次申请
   - 历史申请数据无法按届次归档

3. **数据范围错误**
   - 如直接同步全量32,039条记录
   - 会创建远超需求的账号数量
   - 扩大系统攻击面和管理成本

**根本原因：**
- XG用户API返回全校所有用户，未提供毕业批次筛选
- 缺少"学籍状态"或"毕业批次"字段

**缓解措施：**
- ✓ 已实施：CSV导入时明确标记is_graduating
- ✓ 已实施：API不创建新用户，不覆盖is_graduating
- ⚠️ 风险：每届毕业需手工更新CSV标识

---

#### P0-3: graduation_year（毕业年份）缺失

**不满足表现：**
- XG API未提供graduation_year或预计毕业时间
- 无法从其他字段推导（grade字段也缺失）
- 32,039条记录无一包含毕业年份信息

**业务影响：**
1. **数据归档失败**
   - 无法按毕业年份归档历史数据
   - 多届毕业生数据混杂
   - 历史查询和统计失去时间维度

2. **业务规则失效**
   - 某些时限计算可能依赖毕业年份
   - 无法验证"应届毕业生"身份
   - 跨年度批次对比分析不可行

3. **报表统计缺失维度**
   - 年度统计报表无法生成
   - 趋势分析缺少关键字段
   - 审计追溯困难

**根本原因：**
- XG用户API未设计学籍周期相关字段
- 可能需要教务系统学籍API补充

**缓解措施：**
- ✓ 已实施：CSV导入维护graduation_year
- ✓ 已实施：API不覆盖graduation_year字段
- ⚠️ 限制：年份更新需手工维护CSV

---

#### P0-4: 班级-辅导员映射关系缺失

**不满足表现：**
- XG用户API不提供class_id到counselor_employee_id的映射
- 无法从用户记录推导班级与辅导员关系
- 需要独立数据源维护ClassMapping表

**业务影响：**
1. **审批链路中断**
   - 即使有class_id，也无法查找对应辅导员
   - `class_id → counselor_id` 映射查询失败
   - 学生申请无法进入辅导员审批环节

2. **权限管理失效**
   - 辅导员无法查看所属班级学生申请
   - 跨班级审批阻断逻辑无法实现
   - 审批权限范围划分失败

**根本原因：**
- XG用户API只提供用户维度数据
- 班级-辅导员关系属于组织架构数据，需独立接口

**缓解措施：**
- ✓ 已实施：CSV导入ClassMapping表
- 建议：向教务/人事系统申请班级辅导员关系接口

---

#### P0-5: 宿舍清退状态不在API范围

**不满足表现：**
- XG用户API不包含宿舍清退相关字段
- 无checkout_status、checkout_date等信息
- 属于宿舍管理系统业务数据，非用户基本信息

**业务影响：**
1. **提交前置条件无法验证**
   - 系统要求"已完成宿舍清退"才能提交申请
   - 无法实时查询清退状态
   - 阻断逻辑无法自动执行

2. **降级方案依赖人工**
   - 必须允许学生上传宿舍清退证明截图
   - 需要管理员人工审核
   - 增加运营成本和审批时长

**根本原因：**
- 宿舍清退数据属于宿舍管理系统
- 需要独立宿舍API或系统间数据对接

**缓解措施：**
- 建议：对接宿舍管理系统清退状态API
- ✓ 已实施：降级方案支持上传证明截图

### 3.2 P1级别：实现质量问题（影响功能完整性）

#### P1-1: mapper不支持实际user_identity对象结构

**不满足表现：**
- XG API实际返回：`{"id": 4, "name": "学生"}`（对象结构）
- 当前mapper假设：`'1'` 或 `'student'`（字符串值）
- 实际学生会被判定为 `unknown_user_identity` 并跳过

**代码位置：**
- `backend/apps/users/integrations/xg_user_mapper.py:41, 59-67`

**影响：**
- 实际API同步时，所有学生可能被跳过
- 同步成功率为0%
- 功能完全失效

**修复建议：**
- 支持对象结构：`user_identity.name == "学生"`
- 记录并测试 `id=4` 的含义
- 添加基于真实结构的单元测试

---

#### P1-2: mapper不支持department数组结构

**不满足表现：**
- XG API实际返回：`[{"name": "计算机学院", "level": 2}]`（数组）
- 当前mapper：直接赋值 `department = xg_user.get('department')`
- 可能将列表写入字符串字段，导致类型错误

**代码位置：**
- `backend/apps/users/integrations/xg_user_mapper.py:44, 81`

**影响：**
- 数据库写入失败或类型错误
- department字段值错误（显示为"[{'name':...}]"）

**修复建议：**
- 取 `department[0].name`
- 或按level层级选择
- 添加空数组容错测试

---

#### P1-3: phone字段20%为空字符串

**不满足表现：**
- 20条采样中，4条phone为空字符串`""`
- 非NULL但无效值

**影响：**
- 联系信息不完整
- 通知发送失败率上升
- 用户体验下降

**修复建议：**
- 空字符串归一化为NULL
- API同步时跳过空值，不覆盖已有有效值

## 五、推荐对接策略

### 5.1 Phase 1：CSV主导 + XG API补充（当前推荐）

**适用场景：**
- ✓ XG API缺少关键业务字段
- ✓ 数据变更频率低（每学期1-2次）
- ✓ 对数据时效性要求不高

**数据流：**
```
第1步：CSV导入（主数据源）
  ↓ 创建用户
  ↓ 包含：student_id, name, class_id, is_graduating, graduation_year
  ↓ 包含：major, grade, class_name
  ↓ 状态：active=true

第2步：XG API同步（补充信息源）
  ↓ 匹配已存在用户（按student_id）
  ↓ 更新字段：phone, department, active/status
  ↓ 不创建新用户
  ↓ 不覆盖：class_id, is_graduating, graduation_year
```

**执行规则：**

**允许：**
- ✓ CSV创建维护所有用户主数据
- ✓ CSV维护ClassMapping班级-辅导员映射
- ✓ XG API匹配本地已存在学生
- ✓ XG API更新phone/department/active
- ✓ 对空值、缺失对象容错
- ✓ 输出dry-run报告

**禁止：**
- ✗ XG API创建新学生
- ✗ XG API覆盖class_id/is_graduating/graduation_year
