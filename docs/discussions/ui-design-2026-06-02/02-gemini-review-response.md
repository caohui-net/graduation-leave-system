## 1. UI设计方案

根据需求分析和提供的参考图片，提出以下整体UI设计方案。

### 1.1 设计原则与风格指南
*   **极简卡片化 (Card-based Design)：** 以白色卡片作为内容的承载单元，配合浅灰色（`#F5F5F5`或`#F6F7F9`）背景色，形成清晰的视觉层级。这非常契合微信小程序的原生风格。
*   **一致的主题色 (Consistent Theme)：** 根据参考图片和常见的企业/校园系统应用，建议采用经典的科技蓝/可信蓝作为主色调（Primary Color: `#1890FF` 或 `#2B79F5`）。用于主要按钮、Tab选中态、关键信息高亮。
*   **清晰的字体层级 (Typography)：** 
    *   标题/重要信息：16px (32rpx), 加粗, 深灰 (`#333333`)
    *   正文/常规信息：14px (28rpx), 常规, 中灰 (`#666666`)
    *   辅助说明/时间：12px (24rpx), 常规, 浅灰 (`#999999`)
*   **状态可视化 (Status Visibility)：** 审批状态（待审批、已通过、已驳回）需使用不同颜色标签（Tag）进行区分。

### 1.2 颜色主题与配色方案
*   **主色调 (Primary)：** `#1890FF` (蓝色，用于主按钮、Active态)
*   **背景色 (Background)：** `#F6F7F9` (页面底色)
*   **卡片底色 (Surface)：** `#FFFFFF`
*   **边框色 (Border)：** `#EEEEEE` 或 `#E8E8E8` (用于分割线、输入框边框)
*   **文本色 (Text)：**
    *   主标题/强调文本：`#333333`
    *   副标题/常规文本：`#666666`
    *   辅助/占位符文本：`#999999`
*   **状态色 (Status)：**
    *   待处理 (Warning/Pending)：`#FAAD14` (橙色)
    *   已通过 (Success)：`#52C41A` (绿色)
    *   已驳回 (Error/Reject)：`#FF4D4F` (红色)

### 1.3 页面组件细节与布局

#### A. 学生申请页面
*   **顶部用户信息区：** 采用卡片包裹。当前只有姓名和学号，**建议扩展**：如果后端能提供学院、专业、班级信息，应一并展示，呈现为“标签-值”对齐的列表。
*   **表单区：** 
    *   输入框、选择器不使用全包围边框，而是采用**下划线样式**或**浅灰色填充背景无边框**样式，显得更加轻量。
    *   离校原因的 `textarea` 保持合理的默认高度，并给出字数统计（当前已实现，需微调样式）。
*   **提交按钮：** 底部悬浮固定或跟随页面滚动，大圆角（`border-radius: 44rpx`），通栏蓝色。

#### B. 审批列表页面
*   **顶部导航 Tab：** 
    *   均分宽度，底部带有滑动指示条（蓝色下划线）。
    *   选中状态文字加粗，颜色变为 `#1890FF`。
*   **列表项卡片 (List Item)：**
    *   结构：[顶部区] (申请人名字 + 状态标签) -> [中间区] (申请详情，如离校日期、原因截取) -> [底部区] (申请时间 + 查看详情提示)。
    *   状态标签需要做成带圆角的块状，背景色浅，文字色深（例如待审批：背景 `#FFFBE6`，文字 `#FAAD14`）。

#### C. 详情页面
*   **信息展示卡片：** 分为“基本信息”和“附加信息”模块，采用“左灰色标签，右黑色内容”的对齐布局。
*   **审批流程时间线 (Timeline)：**
    *   设计为左侧一条竖线，带有圆点节点。
    *   节点状态区分：当前节点（蓝色高亮），已完成节点（绿色勾选/置灰）。
    *   右侧展示：审批步骤名称、审批人、时间、审批意见（如有）。
*   **底部操作区：** 对于审批人，“通过”和“驳回”按钮并排显示，建议“驳回”使用描边按钮（白底红字/白底黑字），“通过”使用实心蓝底按钮。

## 2. 技术实现建议

为了遵循“保守原则”并降低依赖风险，建议**不引入**外部UI库（如Vant Weapp、ColorUI等），全部使用原生WXML/WXSS实现。

### 2.1 关键组件实现方案

*   **Tab切换 (Tabs)：**
    *   使用原生的 `view` + `flex` 布局实现。
    *   通过 `wx:for` 渲染Tab列表，点击时更新 `data.currentTab`。
    *   WXSS中使用伪元素 `::after` 实现选中时的底部蓝线指示器。
*   **时间线 (Timeline) - 详情页：**
    *   结构：外层列表包含每一项 `timeline-item`。
    *   布局：使用 `flex`，左侧为时间轴图标（圆圈+下划线），右侧为内容区。
    *   CSS技巧：最后一个 `timeline-item` 的左侧竖线需要隐藏（使用 `:last-child` 伪类）。
*   **状态标签 (Status Tag)：**
    *   在WXSS中定义一系列预设类：`.tag-pending`, `.tag-approved`, `.tag-rejected`。
    *   在WXML中通过数据绑定动态添加类名：`class="tag {{'tag-' + item.status}}"`。

### 2.2 WXSS样式组织策略
*   **全局样式 (`app.wxss`)：** 提取公用的背景色、卡片统一样式（`.card`）、按钮统一样式（`.btn-primary`, `.btn-outline`）、字体颜色类、Flex布局辅助类（如 `.flex-row`, `.justify-between`）。
*   **响应式单位：** 全面使用 `rpx` (Responsive Pixel) 作为长度和字体单位，以确保在不同手机屏幕（不同宽度）下的自适应缩放（如 `margin: 32rpx; font-size: 28rpx;`）。目前 `student-application.wxss` 中混用了 `px`，**需要重构为 `rpx`**。

### 2.3 性能优化建议
*   对于列表页，由于审批记录可能较多，应确保已有分页逻辑（如果有的话）。如果没有，建议在UI层面上实现一个骨架屏（Skeleton Screen）或使用现有的加载提示（Loading），避免白屏。
*   列表图片/头像（如有）：使用懒加载 `lazy-load="true"`。

## 3. 实施计划建议

建议分为三个阶段实施，总体预估工作量4-5小时（比原预估略微充足，以确保代码质量和自测）。

### Phase 1: 基础重构与审批列表页 (优先级：高)
**理由：** 审批列表是高频使用的入口页面，先统一全局变量和组件基础，能为后续页面铺平道路。
**预估工作量：** 1.5 - 2 小时
1.  修改 `app.wxss`，加入全局卡片、颜色、排版基础类。将硬编码的 `px` 换算为 `rpx` (1px ≈ 2rpx)。
2.  实现原生的 Tab 切换栏样式。
3.  重构 `approvals.wxml` 和 `approvals.wxss` 中的列表项，实现卡片化和状态标签。
4.  优化空状态 (`empty`) 的视觉表现。

### Phase 2: 学生申请表单页 (优先级：中)
**理由：** 现有表单已具备基础结构，主要是“穿衣服”的优化工作。
**预估工作量：** 1 - 1.5 小时
1.  重构 `student-application` 的样式，应用 `app.wxss` 中定义的 `.card` 样式。
2.  优化表单项布局，去除粗糙的边框，改用更现代的表单行样式。
3.  优化底部提交按钮（全宽固定悬浮或大圆角按钮）。

### Phase 3: 详情页与时间线实现 (优先级：中)
**理由：** 详情页结构最复杂，尤其是需要新增时间线组件，放最后实现可以复用前两个阶段的卡片和按钮样式。
**预估工作量：** 1.5 - 2 小时
1.  将详情信息分组放入卡片模块。
2.  **核心开发：** 实现纯CSS的原生审批流程时间线 (`Timeline`)。
3.  优化底部审批操作区的按钮并排布局 (`flex` 布局）。
4.  优化附件列表的外观。

## 4. 风险和问题识别

### 4.1 潜在的技术与实施风险
*   **数据结构差异：** UI设计要求展示“学院、专业”等信息，但当前查看 `student-application.wxml` 中似乎只绑定了 `userName` 和 `userId`。如果后端接口尚未返回这些额外字段，UI实现只能先放置占位符，需确认后端进度或只展示现有字段。
*   **状态映射 (Status Mapping)：** 后端返回的 `status` 和 `decision` 字段枚举值必须清晰（例如：pending, approved, rejected），如果目前是数字或其他不易阅读的格式，前端可能需要增加映射字典。
*   **时间线数据支持：** 详情页中的审批历史时间线需要后端提供数组格式的审批节点记录。当前 `detail.approvals` 已有这个结构，但需要确认其是否包含按时间正序/倒序排列的完整节点。

### 4.2 改进建议
*   **交互反馈 (Feedback)：** 按钮点击需要有明显的按压态反馈（例如使用 `hover-class`）。
*   **表单验证可视化：** 当前错误信息展示在底部，建议可以结合输入框边框变红或行内提示的方式，提升表单交互体验。

### 4.3 与现有实现的兼容性考虑
*   当前 `approvals.ts`, `detail.ts`, `student-application.ts` 的业务逻辑（如网络请求、数据绑定）无需修改。
*   我们主要是替换 `.wxml` 中的标签结构，并彻底重写 `.wxss`，这是一种侵入性较低、安全性较高的做法。只要不改变 `bindtap` 绑定的函数名和双向绑定的 `{{}}` 变量名，就不会破坏现有逻辑。
