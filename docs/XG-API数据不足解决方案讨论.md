# XG API数据不足问题解决方案讨论

**问题：** XG API缺失class_id/is_graduating/graduation_year等关键字段  
**影响：** 无法独立支持毕业生离校申请审批系统  
**讨论目标：** 确定可行解决方案  
**日期：** 2026-06-03

---

## 一、问题定义

### 1.1 核心缺失

| 缺失项 | 影响 | 是否可绕过 |
|-------|------|----------|
| class_id | 审批人分配失败 | ❌ 不可绕过 |
| is_graduating | 申请资格判定失败 | ❌ 不可绕过 |
| graduation_year | 数据归档失败 | ⚠️ 可降级 |
| ClassMapping | 审批链路中断 | ❌ 不可绕过 |

### 1.2 问题根源

**技术层面：**
- XG用户API设计未包含学籍周期信息
- XG用户API不提供组织架构关系

**业务层面：**
- 数据分散在不同系统（教务/人事/学工）
- 缺少统一学生信息接口

---

## 二、解决方案对比

### 方案A：CSV主导 + XG API补充（推荐，当前实施）

**实施方式：**
```
CSV导入 → 创建用户 + 全部必填字段
         → 创建ClassMapping
XG API  → 仅更新已存在用户的phone/department
```

**优点：**
- ✓ 业务完整性100%保障
- ✓ 数据可控，可预校验、可回滚
- ✓ 实施周期短（1-2周）
- ✓ 风险低，API故障不影响核心业务
- ✓ 成本可控（每学期更新1-2次）

**缺点：**
- ⚠️ 数据维护成本（需手工整理CSV）
- ⚠️ 时效性差（phone变更需等同步周期）
- ⚠️ 重复维护（CSV+API双数据源）

**适用场景：**
- ✓ 当前阶段（XG API未扩展字段）
- ✓ 数据变更低频（每学期1-2次）
- ✓ 用户规模可控（约1000人）

**实施成本：** 低  
**技术风险：** 低  
**业务风险：** 低  

**评分：85/100** ⭐⭐⭐⭐

---

### 方案B：请求XG API字段扩展

**实施方式：**
向XG系统提交需求，扩展用户API返回：
- 增加`class_id`字段
- 增加`is_graduating`字段  
- 增加`graduation_year`字段
- 提供班级-辅导员关系接口

**优点：**
- ✓ 根本解决问题
- ✓ 数据源统一，时效性高
- ✓ 减少CSV维护成本
- ✓ 后续可切换到API主导模式

**缺点：**
- ✗ 实施周期长（需XG开发、测试、上线）
- ✗ 不可控（依赖外部系统排期）
- ✗ 可能被拒绝（不属于XG核心需求）
- ⚠️ 字段定义需协调（班级ID格式、毕业批次规则）

**适用场景：**
- 长期规划（2026年秋季及以后）
- XG系统有扩展计划
- 学校层面协调支持

**实施成本：** 无（依赖XG）  
**技术风险：** 中（字段定义、数据质量）  
**业务风险：** 高（排期不可控）  

**评分：60/100** ⭐⭐⭐

**结论：** 可作为长期方向，但短期不可行。

---

### 方案C：对接教务系统API

**实施方式：**
绕过XG，直接对接教务系统获取：
- 学生学籍信息（含class_id/grade/major）
- 毕业批次信息（is_graduating/graduation_year）
- 班级管理信息（ClassMapping）

**优点：**
- ✓ 数据权威性高（教务系统为源头）
- ✓ 字段完整（包含学籍周期信息）
- ✓ 时效性好（实时同步）

**缺点：**
- ✗ 需要单独对接（开发、测试、维护成本）
- ✗ 可能无API或API不开放
- ✗ 数据格式未知（需重新适配）
- ⚠️ 多系统依赖（XG+教务，故障点增加）

**适用场景：**
- 教务系统有开放API
- 学校IT部门支持对接
- 对数据时效性要求高

**实施成本：** 中高（新接口开发）  
**技术风险：** 中（多系统集成）  
**业务风险：** 中（依赖教务系统稳定性）  

**评分：65/100** ⭐⭐⭐

**结论：** 可行但成本高，需评估教务API可用性。

---

### 方案D：调整业务逻辑以适配现有数据

**实施方式：**
修改系统设计，绕过缺失字段：

**变通1：** 手工指定审批人（放弃自动分配）
- 学生提交时选择辅导员
- 或管理员手工分配

**变通2：** 全员开放申请权限（放弃资格判定）
- 不验证is_graduating
- 允许所有用户提交

**变通3：** 不按年份归档（放弃批次管理）
- 所有申请混合存储
- 报表时按提交时间统计

**优点：**
- ✓ 无需等待外部系统
- ✓ 实施快速

**缺点：**
- ✗ 严重降低业务价值
- ✗ 增加用户操作负担
- ✗ 系统安全性下降（无资格验证）
- ✗ 数据质量差（错误申请、垃圾数据）

**适用场景：**
- 无其他方案可选
- 业务容忍度极高

**实施成本：** 低  
**技术风险：** 低  
**业务风险：** 极高  

**评分：30/100** ⭐

**结论：** ❌ 不推荐。核心业务逻辑受损，不可接受。

---

### 方案E：混合方案（CSV核心 + 多API补充）

**实施方式：**
```
CSV        → class_id, is_graduating, graduation_year（核心必填）
XG API     → phone, department, active（补充信息）
教务API    → major, grade（可选，如果可用）
宿舍API    → checkout_status（独立接口）
```

**优点：**
- ✓ 各取所长（CSV保障核心，API补充细节）
- ✓ 灵活扩展（后续可逐步接入更多API）
- ✓ 降级容错（单个API故障不影响整体）

**缺点：**
- ⚠️ 复杂度高（多数据源协调）
- ⚠️ 维护成本高（多个同步任务）
- ⚠️ 数据一致性风险（多源冲突）

**适用场景：**
- 多个系统部分可用
- 对数据完整性和时效性都有要求
- 有充足开发资源

**实施成本：** 高  
**技术风险：** 中  
**业务风险：** 低  

**评分：70/100** ⭐⭐⭐

**结论：** 可行，适合长期演进，但初期过于复杂。

## 五、风险与应对

### 5.1 方案A实施风险

**风险1：CSV数据质量问题**
- **表现：** class_id填写错误、重复、格式不一致
- **影响：** 审批人分配失败
- **应对：** 导入前强校验 + 错误报告 + 修正流程

**风险2：每学期手工维护成本**
- **表现：** 需要整理CSV，操作繁琐
- **影响：** 运维负担重
- **应对：** 编写自动化脚本 + 操作手册 + 培训

**风险3：数据时效性差**
- **表现：** phone变更后需等下次CSV更新
- **影响：** 通知失败率上升
- **应对：** 允许学生自行修改phone + XG API定期同步
