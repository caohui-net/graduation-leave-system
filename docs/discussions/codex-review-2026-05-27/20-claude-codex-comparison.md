## 共识点

### 两份分析都发现的问题

1. **Token有效期过长**
   - Codex: "Token 有效期设计偏宽，7天偏长"
   - Claude: "JWT Token 7天有效期过长，建议1小时"
   - **共识：** 缩短为1小时

2. **性能目标不一致**
   - Codex: "计划1000+，设计500，应统一"
   - Claude: "500并发不现实，应为100 QPS"
   - **共识：** 统一为现实目标（100 QPS）

3. **工作日计算需要统一**
   - Codex: "24小时 vs 8小时工作时间，应统一"
   - Claude: "60+行复杂逻辑，建议简化为24小时"
   - **共识：** 简化为24小时提醒

4. **文档口径需要统一**
   - Codex: "多数据库支持残留，应修正"
   - Claude: "SQLAlchemy备选方案应删除"
   - **共识：** 统一为PostgreSQL单数据库 + API集成

## 需要讨论的争议点

### 争议1：applications_history表是否保留？

**保留理由：**
- 完整快照便于回溯
- 审计合规可能需要

**删除理由：**
- audit_logs已有字段级追踪
- 存储冗余
- 查询困难

**建议方案：**
- 只在关键节点创建快照（提交、最终通过）
- 或完全依赖audit_logs

### 争议2：微信绑定安全措施保留几项？

**Codex倾向：** 保留5项（文档已完成）

**Claude倾向：** 简化为2项（实施成本）

**5项措施：**
1. 学生身份验证（短信/邮件/学生证）
2. 受限Token（scope限制）
3. 事务锁（select_for_update）
4. 审计日志
5. 通用错误消息

**建议方案：**
- **Phase 1保留2项**：密码验证 + 审计日志
- **Phase 2按需添加**：如果发现安全问题，再引入其他措施

### 争议3：Celery是否Phase 1引入？

**设计文档：** 已包含Celery

**Claude建议：** Phase 1不引入，先同步实现

**建议方案：**
- Phase 1同步实现通知发送
- 监控性能
- 如果成为瓶颈，Phase 2引入Celery
