## 二、变更类型与传播规范

**所有变更必须按类型走对应的传播路径，不得混用。**

### 2.1 代码变更
```
199开发 → (rsync inotify自动) → 196 staging → (promote-to-prod.sh手动) → 196 production
```
- 现状：✅ 已有机制，保持现状

### 2.2 Schema Migration（表结构变更）
**规范：** 每次代码部署后自动执行 `python manage.py migrate --noinput`

- 生产：✅ `promote-to-prod.sh` 已加入（本次修复）
- Staging：❌ 需在 staging 重启脚本中加入（见 P2 修复）

### 2.3 Data Migration（数据变更）
**规范：** 任何数据变更必须写成 Django `RunPython` migration，禁止手动 SQL

```python
# 正确示例：backend/apps/<app>/migrations/XXXX_描述.py
class Migration(migrations.Migration):
    operations = [
        migrations.RunPython(your_function, migrations.RunPython.noop),
    ]
```

**判断标准：** 凡是在开发环境执行过的 SQL（增删改数据），都必须转化为 migration 文件提交到代码库。

### 2.4 服务依赖变更（requirements/Docker）
**规范：** requirements 中不得使用本地绝对路径；依赖变更后必须重建容器镜像

- biz-core 问题暂缓，正式集成时需将路径改为可在容器内解析的方式

### 2.5 配置变更（.env）
**规范：** 生产配置通过 196 上的专用文件管理，不硬编码在 `promote-to-prod.sh` 中

## 三、修复方案

### P2：Staging 重启加入 migrate

**位置：** 196 的 staging docker-compose 或重启脚本

**修复：** 在 staging 服务重启后执行 migrate：
```bash
# 在 staging 后端容器重启后执行
docker exec staging-graduation-backend python manage.py migrate --noinput
```

或在 staging 的 docker-compose.yml backend service command 中改为：
```yaml
command: >
  sh -c "python manage.py migrate --noinput &&
         python manage.py runserver 0.0.0.0:8000"
```

### P3：Data Migration 规范落地

**操作规范（强制）：**
1. 执行任何 SQL 数据操作前，先确认是否存在对应 migration
2. 开发环境验证后，将等价操作写成 `RunPython` migration 提交
3. Code review 时检查：有数据变更的功能是否配套了 data migration

**模板文件位置：** `backend/apps/<app>/migrations/XXXX_data_描述.py`

### P5：生产 SECRET_KEY

**问题：** `promote-to-prod.sh` 中 SECRET_KEY 是 `production-secret-key-change-in-real-deployment`（明文 placeholder）

**修复：** 在 196 上维护独立的生产配置文件，promote 脚本从该文件读取而非硬编码：
```bash
# /opt/graduation-leave-system/.secrets/production.env（仅196本地，不进代码库）
SECRET_KEY=<真实密钥>
```

promote 脚本改为：
```bash
source /opt/graduation-leave-system/.secrets/production.env
```

### P6：版本标识

**在 /readyz 接口或独立 /version 接口返回当前版本信息：**
```python
# 部署时写入 VERSION 文件（promote 脚本中加一行）
echo "$(date +%Y%m%d%H%M%S)-$(git -C /opt/graduation-leave-system/staging rev-parse --short HEAD 2>/dev/null || echo 'unknown')" \
  > /opt/graduation-leave-system/production/VERSION
```

### P7：Promote 记录

**在 promote-to-prod.sh 末尾追加日志：**
```bash
echo "$(date '+%Y-%m-%d %H:%M:%S') promote SUCCESS staging→production" \
  >> /opt/graduation-leave-system/deploy.log
```
