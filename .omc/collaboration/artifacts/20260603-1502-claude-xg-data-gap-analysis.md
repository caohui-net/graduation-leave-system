## 三、关键缺失字段影响分析

### 3.1 class_id（班级ID）缺失

**影响等级：** 🔴 CRITICAL - 阻断核心业务流程

**业务影响：**
1. **无法自动分配审批人**
   - 学生提交申请时，系统无法通过class_id查找对应辅导员
   - 必须手工指定审批人或使用默认规则

2. **无法建立班级-辅导员映射**
   - ClassMapping表依赖class_id作为主键
   - 无法维护班级与辅导员的对应关系

3. **数据完整性受损**
   - User表的class_id字段（学生必填）无法从API获取
   - 新用户创建时缺少关键业务字段

**缓解措施：**
- ✓ 通过CSV导入维护class_id（已实施）
- ✓ API同步时不覆盖class_id字段（已实施）
- ⚠️ 新用户必须由CSV创建，API无法独立创建学生用户

### 3.2 is_graduating（是否毕业生）缺失

**影响等级：** 🔴 CRITICAL - 影响核心业务逻辑

**业务影响：**
1. **无法判定申请资格**
   - 只有is_graduating=true的学生才能提交离校申请
   - API无法识别哪些学生是当届毕业生

2. **批次管理困难**
   - 系统需要区分不同届次的毕业生
   - 无法通过API自动更新毕业生标识

**缓解措施：**
- ✓ 通过CSV导入维护is_graduating（已实施）
- ⚠️ 每届毕业季需要手工更新CSV标识

### 3.3 graduation_year（毕业年份）缺失

**影响等级：** 🔴 CRITICAL - 影响数据分析和归档

**业务影响：**
1. **数据归档困难**
   - 无法按毕业年份筛选和归档历史数据
   - 报表统计缺少关键维度

2. **业务规则判断**
   - 某些流程可能依赖毕业年份判断（如时限计算）

**缓解措施：**
- ✓ 通过CSV导入维护graduation_year（已实施）

## 五、推荐对接策略

### 5.1 Phase 1：CSV主导 + API补充（推荐，当前实施）

**策略描述：**
- **CSV导入：** 创建所有学生用户，包含完整必填字段
- **XG API：** 仅补充或更新已存在用户的phone/department等非关键字段
- **API不创建新用户**（因缺少class_id/is_graduating/graduation_year）

**数据流：**
```
CSV导入（主） → 创建用户 + class_id + is_graduating + graduation_year
     ↓
XG API同步（补充） → 更新已存在用户的 phone/department
```

**优点：**
- ✓ 业务完整性：所有必填字段通过CSV保证
- ✓ 可控性高：CSV可预校验、可回滚
- ✓ 实施快：无需等待XG API扩展字段
- ✓ 风险低：API故障不影响核心业务

**缺点：**
- ⚠️ 数据维护成本：每学期需更新CSV
- ⚠️ 时效性：phone/department变更需等待API同步周期

**适用场景：**
- ✓ 当前阶段（XG API字段未扩展）
- ✓ 数据变更频率低（每学期1-2次）
- ✓ 对数据时效性要求不高

### 5.2 Phase 2：API主导 + CSV补充（未来可能）

**前提条件：**
- XG API必须提供class_id字段
- XG API必须提供is_graduating字段
- XG API必须提供graduation_year字段
- Live测试确认字段完整性≥95%

**策略描述：**
- **XG API：** 创建和更新所有用户，包含关键业务字段
- **CSV导入：** 仅补充API缺失的字段（如email）

**评估：**
- 🟡 待定 - 需XG API字段扩展后重新评估
- 📅 预计最早实施时间：2026年秋季学期（如API升级）

## 六、风险与缓解措施

### 6.1 数据完整性风险

**风险：** CSV导入时class_id填写错误，导致审批人分配失败

**影响：** 🔴 HIGH - 学生申请无法进入审批流程

**缓解措施：**
- ✓ 已实施：CSV导入强校验class_id存在性
- ✓ 已实施：导入报告显示未匹配班级列表
- 建议：导入前提供class_id预校验工具

### 6.2 数据同步时效性风险

**风险：** 学生手机号变更后，CSV未及时更新，API同步覆盖旧值

**影响：** 🟡 MEDIUM - 通知发送失败，学生体验下降

**缓解措施：**
- ✓ 已实施：API同步只更新非NULL值
- ✓ 已实施：phone字段允许学生自行修改
- 建议：API同步前比对updated_at时间戳，取最新值

### 6.3 API依赖性风险

**风险：** XG API不稳定或返回格式变更，导致同步失败

**影响：** 🟢 LOW - 不影响核心业务（仅补充字段）

**缓解措施：**
- ✓ 已实施：API同步失败不阻断用户操作
- ✓ 已实施：错误日志和告警机制
- ✓ 已实施：Response结构适配（users数组 vs data.data）

### 6.4 角色识别风险

**风险：** user_identity字段值域未完全确认，可能误识别用户角色

**影响：** 🔴 HIGH - 权限混乱，安全风险

**缓解措施：**
- ✓ 已实施：只接受明确的"学生"身份值
- ✓ 已实施：未知身份值一律跳过，记录到日志
- ✓ 已实施：新用户不自动激活，需人工审核
