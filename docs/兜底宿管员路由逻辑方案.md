# 兜底宿管员路由逻辑实现方案

**日期：** 2026-06-06  
**需求：** 为无楼栋数据学生提供兜底宿管员审批

---

## 业务背景

**当前问题：**
- 116名学生缺少楼栋数据（building_name为空）
- 无法按楼栋路由到对应宿管员
- 申请提交失败（404错误：该楼栋宿管员不存在）

**解决方案：**
- 新增宿管员程婷（职工号：92008149）
- 作为"兜底宿管员"，专门负责无楼栋学生的审批
- 不需要绑定特定楼栋

---

## 实现方案

### 方案A：楼栋特殊标记（推荐）

**逻辑：**
1. 程婷的building字段设置为特殊值：`"未分配"`或`"无楼栋学生"`
2. 无楼栋学生提交申请时，将building设置为`"未分配"`
3. 按building路由到程婷

**修改点：**

`backend/apps/applications/views.py:create_application`

```python
def create_application(request):
    # ... 前面代码保持不变 ...
    
    # 获取学生楼栋信息
    building = user.building
    if not building or building.strip() == '':
        # 无楼栋数据，使用特殊标记
        building = '未分配'
    
    # 查找宿管员
    try:
        dorm_manager = User.objects.get(
            role=UserRole.DORM_MANAGER, 
            building=building, 
            active=True
        )
    except User.DoesNotExist:
        return Response({
            'error': {
                'code': 'NOT_FOUND', 
                'message': '该楼栋宿管员不存在',
                'details': {'building': building}
            }
        }, status=status.HTTP_404_NOT_FOUND)
    
    # ... 后续代码保持不变 ...
```

**数据库修改：**
- 程婷的building字段设置为`"未分配"`

**优点：**
- 路由逻辑统一，无需特殊判断
- 清晰标识无楼栋学生
- 易于查询和统计

**缺点：**
- 需要修改程婷的数据记录

---

### 方案B：Fallback查询（灵活）

**逻辑：**
1. 先按building查找宿管员
2. 如果找不到（building为空或不存在），查找兜底宿管员
3. 兜底宿管员的building字段为NULL或空字符串

**修改点：**

`backend/apps/applications/views.py:create_application`

```python
def create_application(request):
    # ... 前面代码保持不变 ...
    
    # 查找宿管员
    building = user.building
    dorm_manager = None
    
    if building and building.strip():
        # 有楼栋信息，按楼栋查找
        try:
            dorm_manager = User.objects.get(
                role=UserRole.DORM_MANAGER, 
                building=building, 
                active=True
            )
        except User.DoesNotExist:
            pass  # 找不到，继续fallback
    
    if not dorm_manager:
        # 无楼栋或找不到对应宿管员，使用兜底宿管员
        try:
            dorm_manager = User.objects.get(
                role=UserRole.DORM_MANAGER,
                user_id='92008149',  # 程婷职工号
                active=True
            )
        except User.DoesNotExist:
            return Response({
                'error': {
                    'code': 'NOT_FOUND', 
                    'message': '无可用宿管员',
                    'details': {'building': building or '未分配'}
                }
            }, status=status.HTTP_404_NOT_FOUND)
    
    # ... 后续代码保持不变 ...
```

**数据库修改：**
- 程婷的building字段保持为NULL或空

**优点：**
- 灵活，兜底逻辑独立
- 不需要特殊building标记
- 易于扩展（可增加多个兜底宿管员）

**缺点：**
- 路由逻辑略复杂
- 硬编码职工号

---

### 方案C：数据库标记位（推荐生产环境）

**逻辑：**
1. User表增加`is_fallback_dorm_manager`字段
2. 程婷设置`is_fallback_dorm_manager=True`
3. 无楼栋学生查找时，先按building查找，失败则查找fallback宿管员

**修改点：**

1. **数据库迁移：**

```python
# migrations/00XX_add_fallback_dorm_manager.py
operations = [
    migrations.AddField(
        model_name='user',
        name='is_fallback_dorm_manager',
        field=models.BooleanField(default=False),
    ),
]
```

2. **路由逻辑：**

```python
def create_application(request):
    # ... 前面代码保持不变 ...
    
    building = user.building
    dorm_manager = None
    
    if building and building.strip():
        try:
            dorm_manager = User.objects.get(
                role=UserRole.DORM_MANAGER, 
                building=building, 
                active=True
            )
        except User.DoesNotExist:
            pass
    
    if not dorm_manager:
        try:
            dorm_manager = User.objects.get(
                role=UserRole.DORM_MANAGER,
                is_fallback_dorm_manager=True,
                active=True
            )
        except User.DoesNotExist:
            return Response({
                'error': {
                    'code': 'NOT_FOUND', 
                    'message': '无可用宿管员'
                }
            }, status=status.HTTP_404_NOT_FOUND)
    
    # ... 后续代码保持不变 ...
```

**优点：**
- 标准数据库设计
- 无硬编码
- 易于维护和扩展
- 支持多个兜底宿管员

**缺点：**
- 需要数据库迁移
- 实现复杂度最高
