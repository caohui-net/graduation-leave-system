```python
def _process_row(self, row, stats, dry_run):
    # 1. 提取字段
    user_id = row['user_id'].strip()
    name = row['name'].strip()
    building = row.get('building_name', '').strip()
    
    # 2. 跳过临时ID
    if user_id.startswith('TMP2026_'):
        stats['skipped'] += 1
        return
    
    # 3. 创建或更新用户
    user, created = User.objects.update_or_create(
        user_id=user_id,
        defaults={
            'name': name,
            'role': UserRole.STUDENT,
            'building': building if building else None,
            'is_graduating': True,
            'graduation_year': 2026,
            # ... 其他字段
        }
    )
```

```python
with transaction.atomic():
```

## 5. 业务流程对数据的操作

### 5.1 数据模型关系图

```
User (用户表)
├── user_id (PK)
├── role: STUDENT | DORM_MANAGER | COUNSELOR | ADMIN
├── department (学院) - 用于辅导员路由
├── building (楼栋) - 用于宿管员路由
└── class_id, phone, email, graduation_year, ...

Application (离校申请表)
├── application_id (PK)
├── student (FK → User) - 申请学生
├── status: DRAFT | PENDING_DORM_MANAGER | PENDING_COUNSELOR | APPROVED | REJECTED
├── contact_phone - 联系电话（必填）
├── reason - 离校原因
└── dorm_checkout_status - 宿舍清退状态

Approval (审批记录表)
├── approval_id (PK)
├── application (FK → Application) - 关联申请
├── step: DORM_MANAGER | COUNSELOR
├── approver (FK → User) - 分配的审批人
├── decided_by (FK → User) - 实际审批人（代理审批时不同）
├── decision: PENDING | APPROVED | REJECTED
└── comment, decided_at

Notification (通知表)
├── notification_id (PK)
├── user (FK → User) - 接收通知的用户
├── type: APPLICATION_SUBMITTED | APPROVAL_PENDING | APPROVED | REJECTED
└── application (FK → Application)
```

### 5.2 申请创建流程（使用导入数据）

```python
# 视图: apps/applications/views.py - ApplicationCreateView
```

### 5.3 审批流程（两级串行审批）

```
Student提交申请
    ↓
status = PENDING_DORM_MANAGER
    ↓
宿管员审批 (基于building路由)
    ├─ APPROVED → status = PENDING_COUNSELOR
    └─ REJECTED → status = REJECTED (结束)
    ↓
辅导员审批 (基于department路由)
    ├─ APPROVED → status = APPROVED (完成)
    └─ REJECTED → status = REJECTED (结束)
```

**路由依赖导入数据**:
- `User.building` → 决定宿管员审批人
- `User.department` → 决定辅导员审批人
- 缺失building → 路由到兜底宿管员（user_id=92008149）

### 5.4 数据生成和更新逻辑

| 操作 | 数据流 | 涉及模型 |
|------|--------|---------|
| **申请创建** | 读取Student的department/building → 生成2条Approval记录 | Application, Approval, Notification |
| **审批操作** | 更新Approval.decision → 更新Application.status → 生成Notification | Approval, Application, Notification |
| **代理审批** | Approval.decided_by ≠ Approval.approver（记录实际审批人） | Approval |
| **宿舍清退** | 调用学工API → 更新Application.dorm_checkout_status | Application（外部API同步） |

## 6. 数据间的逻辑关系

### 6.1 外键约束关系

```sql
-- Application → User (student)
FOREIGN KEY (student_id) REFERENCES users(user_id) ON DELETE PROTECT

-- Approval → Application
FOREIGN KEY (application_id) REFERENCES applications(application_id) ON DELETE PROTECT

-- Approval → User (approver)
FOREIGN KEY (approver_id) REFERENCES users(user_id) ON DELETE PROTECT

-- Approval → User (decided_by)
FOREIGN KEY (decided_by_id) REFERENCES users(user_id) ON DELETE PROTECT

-- Notification → User
FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE

-- Notification → Application
FOREIGN KEY (application_id) REFERENCES applications(application_id) ON DELETE CASCADE
```

### 6.2 业务规则约束

| 规则 | 实现位置 | 说明 |

| **两级串行审批** | `apps/approvals/services.py` | 宿管员审批通过后才能进入辅导员审批 |

#### 6.3.2 运行时一致性

```python
# Application模型的clean()方法
def clean(self):
    active_statuses = [
        ApplicationStatus.PENDING_DORM_MANAGER,
        ApplicationStatus.PENDING_COUNSELOR,
        ApplicationStatus.APPROVED,
    ]
    existing = Application.objects.filter(
        student=self.student,
        status__in=active_statuses
    ).exclude(application_id=self.application_id).exists()
    if existing:
        raise ValidationError('该学生已有待审批或已通过的申请')
```
