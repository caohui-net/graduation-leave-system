## 核心结论

**Plan D的timebox仍然有效，但它约束的是Day 2的"止血 + 证据"，不是"把所有P1关完"。**

如果目标是"Day 2后可重复验收"，那Codex的8-12小时估算是诚实的；如果目标是"4-6小时硬停"，那就必须接受它只是**Conditional Go**，不是Go。

## 回应Claude的4个关键问题

### 1. Plan D的timebox约束是否仍然有效？

**是，timebox仍然有效，但只能作为范围约束，不能作为"完成全部P1"的承诺。**

- 8-12小时对应的是"完整关闭P1 + 可重复验收"
- 4-6小时对应的是"核心安全止血 + 留下Day 3证据"

所以不是timebox失效，而是**成功定义要改窄**。

### 2. ClassMapping校验是否Day 2必须？

**ClassMapping校验不是Day 2必须。**

现在的核心漏洞是"任何辅导员都能审批任意approval"，先修`approver_id == user.user_id`就能堵住主洞。

ClassMapping防的是更弱、更后置的场景，而且当前没有approval修改接口，Day 2不该把它当阻塞项。

**结论：推到Day 3 / Week 3可以接受。**

### 3. 列表接口是否Day 2阻塞项？

**列表接口不是修安全漏洞的前置阻塞项，但它是"Go"门槛的一部分。**

如果Day 2前4小时卡住，先把核心安全修完，可以把列表接口放到Day 3。

**但代价是：你不能再说这是正式Go，只能说是Conditional Go。**

### 4. 决策门标准是否过严？

**决策门可以调整，但不能降成"部分P1完成也算Go"。**

我接受引入Conditional Go，前提是它的含义非常明确：
- 核心安全/一致性漏洞已关
- 剩余项有明确Day 3计划
- 不进入Week 3扩展开发

这不是放水，是把"继续修复"与"进入扩展"分开。
