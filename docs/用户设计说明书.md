## 一、系统概述

### 1.1 系统目的

本系统旨在实现毕业生离校申请的在线化、规范化管理，替代传统纸质审批流程，提高审批效率，减少人工成本。

### 1.2 主要功能

- **学生端：** 在线提交离校申请、上传证明材料、查看审批进度、接收审批通知
- **辅导员端：** 审批本年级学生的离校申请、查看审批历史、接收待办提醒
- **学工部端：** 审批所有学生的离校申请、查看全校审批统计、导出审批数据
- **系统管理：** 用户管理、权限配置、系统参数设置

### 1.3 使用平台

- **移动端：** iOS/Android App（React Native开发）
- **微信小程序：** 微信内使用，无需安装
- **管理后台：** Web浏览器访问（可选）

## 二、业务流程

### 2.1 申请提交流程

```
学生登录 → 填写申请信息 → 上传证明材料 → 提交申请 → 等待审批
```

**必填信息：**
- 学号、姓名、专业、班级
- 计划离校日期
- 联系电话、家庭地址

**必传材料：**
- 宿舍清退证明（照片或PDF）
- 图书馆清书证明（照片或PDF）
- 财务结清截图（照片或PDF）

### 2.2 审批流程

```
学生提交 → 辅导员审批 → 学工部审批 → 审批完成
           ↓（驳回）      ↓（驳回）
         学生修改 ←──────┘
```

**审批时限：**
- 辅导员审批：提交后1个工作日内
- 学工部审批：辅导员通过后1个工作日内
- 超时提醒：系统自动发送微信通知

**审批结果：**
- **通过：** 进入下一审批环节或完成审批
- **驳回：** 学生收到通知，修改后重新提交

### 2.3 通知机制

**微信通知场景：**
- 学生提交申请 → 通知辅导员
- 辅导员审批通过 → 通知学生和学工部
- 辅导员驳回 → 通知学生
- 学工部审批通过 → 通知学生（审批完成）
- 学工部驳回 → 通知学生
- 审批超时 → 通知审批人

## 三、系统特色

### 3.1 双通道认证

**学号密码登录：**
- 适用于首次使用或更换设备
- 学号由学校统一导入
- 初始密码为学号，首次登录强制修改密码
- 密码要求：至少8位，包含字母和数字

**微信授权登录：**
- 绑定后一键登录，无需输入密码
- 支持微信消息推送
- 更换手机后需重新绑定

### 3.2 外部系统对接

**宿舍管理系统对接：**
- 学生提交申请时，系统自动验证宿舍是否已清退
- 验证失败时，提示学生先完成宿舍清退
- 如外部系统不可用，允许手动上传证明

**对接方式：**
- API接口对接（推荐）
- 手动上传证明（降级方案）

### 3.3 数据安全

**传输安全：**
- 全站HTTPS加密传输
- API接口JWT Token认证

**存储安全：**
- 密码加密存储（bcrypt算法）
- 附件文件权限控制（只能查看自己的）
- 审计日志记录（所有操作可追溯）

**隐私保护：**
- 学生只能查看自己的申请
- 辅导员只能查看负责班级的学生
- 学工部可以查看所有学生

- ✓ 提交申请

**辅导员权限：**
- ✓ 查看负责班级学生的申请列表
- ✓ 审批负责班级学生的申请（通过/驳回）
- ✓ 填写审批意见
- ✓ 查看审批历史
- ✓ 接收待审批通知
- ✗ 审批其他班级学生

**学工部权限：**
- ✓ 查看所有学生的申请列表
- ✓ 审批所有学生的申请（通过/驳回）
- ✓ 填写审批意见
- ✓ 查看全校审批统计
- ✓ 导出审批数据（Excel）
- ✓ 查看审批历史

## 五、系统部署

### 5.1 部署方式

**本地部署（推荐）：**
- 部署在学校内网服务器
- 数据存储在学校自有数据库
- 完全掌控数据安全

**部署要求：**
- 服务器：4核CPU、8GB内存、100GB硬盘
- 操作系统：Linux（Ubuntu 20.04+或CentOS 7+）
- 数据库：PostgreSQL 12+
- 容器：Docker 20.10+、Docker Compose 2.0+

### 5.2 系统容量

**设计容量（单实例）：**
- 在线用户：500人
- 并发请求：50个
- 峰值QPS：100次/秒
- 响应时间：<200ms（95%请求）

**数据容量：**
- 学生用户：10,000人
- 年度申请：5,000份
- 附件存储：50GB/年

### 5.3 备份策略

**数据库备份：**
- 每日全量备份（凌晨2点）
- 保留最近30天备份
- 备份文件加密存储

**附件备份：**
- 每周全量备份（周日凌晨）
- 保留最近12周备份

## 六、常见问题

### 6.1 学生常见问题

**Q1：忘记密码怎么办？**
A：联系辅导员或系统管理员重置密码。

**Q2：微信绑定失败怎么办？**
A：确认学号和密码正确，如仍失败请联系管理员。

**Q3：上传附件失败怎么办？**
A：检查文件大小（<10MB）和格式（jpg/png/pdf/doc/docx），如仍失败请联系管理员。

**Q4：申请被驳回后如何修改？**
A：在申请详情页点击"修改"按钮，修改后重新提交。

**Q5：审批进度在哪里查看？**
A：登录后在"我的申请"列表中查看，点击详情可查看审批历史。

### 6.2 审批人常见问题

**Q1：如何查看待审批申请？**
A：登录后在"待审批"列表中查看，点击进入审批页面。

**Q2：审批意见是否必填？**
A：驳回时必须填写驳回原因，通过时可选填。

**Q3：审批后能否撤销？**
A：审批后不能撤销，如需修改请联系系统管理员。

**Q4：如何导出审批数据？**
A：学工部用户在"审批统计"页面点击"导出Excel"按钮。

### 6.3 系统管理问题

**Q1：如何批量导入学生用户？**
A：准备Excel模板（学号、姓名、专业、班级、辅导员），在管理后台上传。

**Q2：如何修改审批时限？**
A：在系统配置页面修改"审批时限"参数（单位：小时）。

**Q3：如何查看系统日志？**
A：在管理后台"审计日志"页面查看，支持按用户、操作、时间筛选。

**Q4：如何对接宿舍管理系统？**
A：参考《数据对接说明文档》，提供API接口或数据导出。

## 六、数据对接说明

### 6.1 为什么需要数据对接

本系统需要从宿舍管理系统获取以下数据，才能正常运行：

1. **学生基本信息** - 创建学生账号，让学生能够登录系统
2. **辅导员基本信息** - 创建辅导员账号，让辅导员能够审批申请
3. **班级-辅导员对应关系** - 系统自动分配审批人，学生提交申请时自动找到对应的辅导员
4. **宿舍清退数据** - 验证学生是否完成宿舍清退，未完成不能提交申请

### 6.2 数据对接方式（简单易懂）

**方式一：Excel文件导入（推荐，最简单）**

适用于：学生信息、辅导员信息、班级-辅导员对应关系

**操作步骤：**
1. 宿管系统导出Excel文件（我们提供模板）
2. 系统管理员登录后台，上传Excel文件
3. 系统自动导入数据，生成账号
4. 导入完成后，学生和辅导员就可以登录了

**优点：**
- 不需要技术开发，1-2周就能上线
- 操作简单，管理员就能完成
- 数据可以先检查再导入，出错了可以重新导入

**更新频率：**
- 每学期导入1-2次即可
- 如果有个别学生或辅导员变动，可以在后台手动修改

---

**方式二：API接口对接（实时查询）**

适用于：宿舍清退数据

**为什么用API：**
- 学生提交申请时，需要实时查询宿舍是否已清退
- 如果未清退，系统会提示学生先完成宿舍清退手续

**如果API不可用怎么办：**
- 允许学生上传宿舍清退证明截图
- 管理员审核通过后，学生可以继续提交申请

### 6.3 需要提供的数据（通俗说明）

**1. 学生基本信息（约1000人）**

需要的字段：
- 学号、姓名、院系、专业
- 班级ID（重要！用于自动分配辅导员）
- 班级名称（用于显示）
- 年级、毕业年份
- 是否为当届毕业生
- 手机号、邮箱（可选，用于找回密码）

**说明：**
- 只需要当届毕业生的数据，不需要全部学生
- 班级ID必须唯一，比如"CS2022-01"，不能重复

---

**2. 辅导员基本信息（约50人）**

需要的字段：
- 工号、姓名、院系
- 手机号、邮箱（可选）
- 是否在职

**说明：**
- 工号必须唯一
- 离职的辅导员标记为"不在职"，不删除记录

---

**3. 班级-辅导员对应关系（约50-100条）**

需要的字段：
- 班级ID（如CS2022-01）
- 班级名称（如计算机科学与技术2022级1班）
- 辅导员工号（如T2022001）

**说明：**
- 一个班级只能有一个辅导员（首版限制）
- 如果辅导员调岗，更新这个表即可

---

**4. 宿舍清退数据（实时查询）**

需要的字段：
- 学号、姓名
- 清退状态（已完成/进行中/未开始）
- 清退完成时间

**说明：**
- 通过API接口实时查询
- 如果API不可用，允许学生上传截图

### 6.4 数据导入顺序（重要）

**必须按以下顺序导入：**
1. 先导入辅导员信息
2. 再导入班级-辅导员对应关系
3. 最后导入学生信息

**为什么要按顺序：**
- 班级映射需要验证辅导员工号是否存在
- 学生导入需要验证班级是否有对应的辅导员
- 如果顺序错了，系统会提示错误

### 6.5 首次导入后的效果

**学生端：**
- 学生可以用学号登录（初始密码需要设置）
- 登录后可以提交离校申请
- 系统自动分配对应的辅导员审批

**辅导员端：**
- 辅导员可以用工号登录
- 登录后可以看到负责班级的学生申请
- 可以进行审批操作

**学工部端：**
- 学工部管理员手动创建（约5人）
- 可以看到所有学生的申请
- 可以进行终审操作

### 6.6 常见问题

**Q1：数据导入会不会覆盖原有数据？**

A：不会。系统采用"智能更新"策略：
- 已存在的学号/工号：更新信息
- 新增的学号/工号：创建新账号
- 本次未导入的学号/工号：标记为停用（不删除）

**Q2：导入错误了怎么办？**

A：可以重新导入。系统会先校验数据，发现错误会提示，不会直接导入。

**Q3：如何知道导入是否成功？**

A：系统会生成导入报告，显示：
- 成功导入多少条
- 失败多少条（包含错误原因）
- 新增多少个账号
- 更新多少个账号
- 停用多少个账号

**Q4：数据多久更新一次？**

A：建议每学期更新1-2次。如果有个别变动，可以在后台手动修改。

**Q5：需要技术人员配合吗？**

A：Excel导入不需要技术人员，管理员就能操作。API接口需要技术人员配合开发。
