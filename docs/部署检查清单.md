## Staging部署步骤

### 1. 数据库迁移（dry-run）
```bash
cd backend
source venv/bin/activate
python manage.py migrate --settings=config.staging --dry-run
```

### 2. 执行迁移
```bash
python manage.py migrate --settings=config.staging
```

### 3. 验证字段
```sql
\d applications;
-- 确认字段存在：
-- application_type, stay_start_date, stay_end_date, stay_reason
```

### 4. 部署代码
```bash
# 后端
git pull origin main
systemctl restart graduation-backend-staging

# 前端
cd frontend
npm run build

```

## 数据库账号规范（生产环境）

### 账号隔离要求
- **应用账号** (`app_prod_user`): 仅DML权限（SELECT/INSERT/UPDATE/DELETE）
- **迁移账号** (`migration_user`): DDL权限（仅发布时使用）
- **只读账号** (`readonly_user`): 仅SELECT权限（监控/报表）

### 权限检查
```sql
-- 验证应用账号无DDL权限
SELECT has_table_privilege('app_prod_user', 'users', 'CREATE');
-- 应返回 false
```

## 回滚预案

### 代码回滚
```bash
# 回滚到上一个稳定版本
git checkout <last_stable_commit>
systemctl restart graduation-backend-prod
```

### 数据库回滚
```bash
# 恢复备份
psql graduation_prod < backup_prod_YYYYMMDD_HHMMSS.sql

# 或执行回滚SQL
psql graduation_prod < rollback_migrations.sql
```

### 触发条件
- 错误率 >5%
- 核心功能不可用
- 数据库死锁/阻塞
- 用户无法登录/提交申请

## 治理框架核心原则

1. **所有数据修复必须Migration化** - 不允许手动UPDATE/DELETE
2. **生产操作必须备份+审计** - 记录WHO/WHAT/WHEN
3. **发布前强制检查** - CI自动拦截未应用的migrations
4. **权限最小化** - 生产应用账号禁止DDL
