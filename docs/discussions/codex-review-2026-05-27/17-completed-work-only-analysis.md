### 设计成果优点

1. 架构选择务实：Django 单体 + 单实例部署符合校园内部系统规模。
2. 数据库方向已收敛：PostgreSQL 作为本项目唯一数据库，避免多数据库运行时兼容成本。
3. 安全设计有明确增强项：微信绑定事务锁、通用错误、审计日志、上传 MIME 校验、API 限流等。
4. 审批流程具备状态机意识：草稿、待辅导员、待学工部、通过、驳回等状态明确。
5. 测试策略比一般设计文档更具体：包含 PostgreSQL 测试、并发绑定测试、限流测试、文件安全测试。

### 已完成设计中的问题

这些问题仅针对已经写入的设计文档本身：

1. **SQL 示例与 PostgreSQL 口径不一致**

   文档声明本项目使用 PostgreSQL，但表结构示例仍大量使用 MySQL 风格语法，例如 `AUTO_INCREMENT`、`COMMENT`、表级 `COMMENT='...'`、内联 `INDEX`、`ON UPDATE CURRENT_TIMESTAMP`。

   如果这些只是概念示例，应标注“伪 SQL，以 Django Model 为准”。如果作为实际 DDL，则需要改成 PostgreSQL 语法。

2. **外键约束有不可执行组合**

   例如字段声明 `NOT NULL`，外键又写 `ON DELETE SET NULL`。这在数据库层面互相冲突。审批记录、审计日志这类历史数据建议优先使用软删除或 `PROTECT/RESTRICT`，避免历史记录失真。

3. **第 6 章存在合并残留**

   外部系统集成章节中，有一段缩进错乱的 Python 代码残留在 SQL 配置示例后，并且章节编号重复。这是已完成文档的结构问题，建议清理。

4. **工作日时限口径需统一**

   文档同时出现“1个工作日(24小时)”和“1个工作日 = 8小时工作时间”的说法。既然已采用 9:00-17:00 工作时间计算，应统一为 8 个工作小时。

5. **Token 有效期设计偏宽**

   Access Token 设计为 7 天。对审批和个人信息系统而言，这个时长偏长。虽然尚未实现，但它属于已完成认证设计的一部分，建议在设计文档阶段调整。
