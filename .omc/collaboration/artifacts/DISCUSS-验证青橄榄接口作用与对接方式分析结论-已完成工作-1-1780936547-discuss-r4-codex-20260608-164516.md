## 二、核心发现与分析

### 2.1 Credentials映射规则

经过3种方案测试,确认正确映射:
- **Header.appKey** = AppKey (`abc0a32aa8dd94d1f765841abaafd8ba`)
- **签名计算** = 使用AppSecret (`b1d2efa9587446d80ce6388e0c0b25131b8dea59`)
- **AppId** (`c6qgh2`) = 暂未在API中直接使用


### 2.2 接口用途分类

**类型1: 批量数据同步接口**
- 接口: 信息中心 `/api/open-api/user-center/tenant/auth-user-info`
- 特点: 一次返回大量数据(32060条), 分页查询
- 用途: 定期同步全量用户数据到本地数据库
- 调用场景: 定时任务(如每天凌晨同步一次)

**类型2: 实时登录认证接口**
- 接口: 移动端 `get_user_code_by_token` + `get_user_info`
- 特点: 单用户实时查询, OAuth流程
- 用途: 用户通过青橄榄移动端跳转登录时验证身份
- 调用场景: 用户访问系统时实时调用

**类型3: 管理员验证接口**
- 接口: 管理端 `verify_admin_user`
- 特点: Token验证, 获取管理员信息
- 用途: 管理员通过青橄榄管理端登录时验证身份
- 调用场景: 管理员访问系统管理功能时调用

## 四、建议的集成方案

### 方案A: 双轨并行(推荐)

**架构:**
```
1. 定时任务(每日凌晨): 信息中心API → 全量同步 → 本地user表
2. 用户登录: 青橄榄SSO → 验证 → 查询本地user表 → 创建session
3. 数据优先级: SSO实时数据 > 本地同步数据
```

**优点:**
- 本地有完整用户数据, 支持离线查询
- 登录时使用实时SSO验证, 数据最新
- 两套系统互为备份

**实现:**
- 新增Django management command: `sync_users_from_infocenter`
- 添加cron任务定期执行
- SSOUserMapping保留, 用于SSO登录时关联

---

### 方案B: 仅SSO实时认证

**架构:**
```
用户登录: 青橄榄SSO → 验证 → 获取用户信息 → 创建/更新本地user → session
不使用信息中心批量同步
```

**优点:**
- 实现简单
- 数据永远是最新的

**缺点:**
- 首次登录才创建本地用户
- 无法提前知道所有用户
- 管理员无法预先分配权限

---

### 方案C: 仅信息中心同步

**架构:**
```
1. 定时同步: 信息中心API → 全量同步 → 本地user表
2. 用户登录: 本地账号密码验证
3. 不使用青橄榄SSO
```

**优点:**
- 实现最简单
- 完全独立, 不依赖青橄榄在线

**缺点:**
- 失去SSO单点登录能力
- 用户需要额外记住本系统密码
- 与项目"青橄榄对接"需求不符
