# ADR 0001: 拒绝实施通用断路器模式

## 状态
已接受 - 2026-06-15

## 背景
毕业生离校系统依赖外部服务（青橄榄SSO、学工API），当前HTTP请求超时配置为30秒。需要评估是否引入断路器（Circuit Breaker）模式来防止级联失败。

系统规模：
- 总用户数：~6000（2026届预计毕业生5675人）
- 并发峰值：可预测且有界
- 架构：Django单体应用 + PostgreSQL

## 决策过程
通过Claude + Codex + Gemini三方技术讨论达成共识（2026-06-15）。

讨论记录：`.collab/artifacts/fast/DISCUSS-断路器决策-6000用户系统-超时30S-1781554606-*`

## 决定
**拒绝实施通用断路器模式。**

采用替代方案：
1. **优化超时配置**（已实施）：HTTP请求超时从30s调整为 `(3, 5)` —— 3秒连接超时 + 5秒读取超时
2. **前端超时控制**：为fetch请求添加AbortController超时机制
3. **监控告警**：外部接口失败计数和告警
4. **条件性断路器**：仅当SSO/信息中心接口出现持续故障且有监控证据时，再考虑针对性引入

## 理由

### 规模不匹配
- 6000用户系统峰值并发低，非微服务/高并发场景
- 企业级弹性模式对此规模系统过度工程

### 超时是根源问题
- 30秒超时在毕业高峰期会导致请求堆积
- Web服务器线程池/连接池可能被耗尽（雪崩效应）
- 快速失败（3-5秒）比断路器更直接有效

### 架构复杂度
- 完整断路器需Redis等分布式状态管理
- 多进程/容器部署增加状态同步复杂度
- 内存存储导致Split-Brain风险
- 误触发风险：高负载≠系统故障

### ROI分析
- 短超时达到断路器90%安全效果
- 维护成本 > 收益
- 过度工程风险 > 级联失败风险（此规模下）

## 推荐超时阈值

| 场景 | 超时配置 | 说明 |
|------|---------|------|
| 实时登录/认证 | 3-5秒 | 连接3秒 + 读取5秒 |
| 批量数据同步 | 10-30秒 | 允许有限重试 |
| 数据库查询 | 不设置 | 本地依赖，无需超时保护 |
| 内部API | 不设置 | 信任内部服务 |

## 适用范围
- ❌ **不适用**：数据库访问、内部CRUD接口、全系统级断路器
- ✅ **适用**：青橄榄SSO、学工API等外部HTTP依赖
- 🔄 **条件性适用**：当外部服务出现"连续失败 + 恢复慢 + 有监控证据"时

## 后果

### 积极影响
- 快速失败防止线程池耗尽
- 架构保持简单，维护成本低
- 避免断路器误触发风险
- 开发和测试工作量小

### 消极影响
- 外部服务故障时无自动熔断
- 依赖监控告警人工介入
- 需定期审查外部依赖可用性

### 缓解措施
- 实施外部接口失败监控告警
- 负载测试验证超时配置有效性
- 保留断路器作为"条件性方案"备选
