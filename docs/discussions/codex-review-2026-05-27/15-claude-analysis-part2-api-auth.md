## 第3章：API设计

### 发现的问题

#### MAJOR - 微信OAuth2流程过于复杂

**问题描述：**
微信登录有3种响应场景：
1. 已绑定 → 直接登录
2. 未绑定 → 返回temp_token，需要绑定
3. 新用户 → 返回limited_token，需要设置密码

**问题：**
- 客户端需要处理3种不同响应格式
- 状态管理复杂（temp_token vs limited_token）
- 用户体验差（多次跳转）

**影响范围：**
- 前端代码复杂
- 用户流程冗长
- 测试用例多

**建议方案：**
**简化为2步流程**：
```
Step 1: POST /api/v1/auth/wechat/login
Response:
{
  "status": "success|need_bind|need_setup",
  "token": "xxx",  // 统一token字段
  "next_action": "/api/v1/auth/wechat/bind" or "/api/v1/auth/password/setup"
}

Step 2: 根据next_action调用对应接口
```

#### MAJOR - API版本控制策略不完整

**问题描述：**
设计提到"URL版本：/api/v1/"和"废弃API保留至少2个版本"，但：
1. **版本升级策略未定义**：何时发布v2？
2. **兼容性测试未提及**：如何确保v1/v2共存？
3. **废弃通知机制缺失**：如何通知客户端升级？

**建议方案：**
1. **版本升级触发条件**：
   - 破坏性变更（字段删除、类型变更）
   - 重大功能重构

2. **废弃流程**：
   ```
   v1.0 → v2.0发布 → v1.0标记deprecated（响应头）
   → 6个月过渡期 → v1.0下线
   ```

3. **响应头通知**：
   ```
   X-API-Version: 1.0
   X-API-Deprecated: true
   X-API-Sunset: 2027-01-01
   ```

#### MINOR - 文件上传返回409冲突码语义不准确

**问题描述：**
文件上传时，如果SHA256哈希重复，返回409 Conflict。但这不是"冲突"，而是"去重"。

**建议方案：**
返回200 OK + 去重信息：
```json
{
  "code": 200,
  "message": "文件已存在，已关联到申请",
  "data": {
    "id": 123,
    "is_duplicate": true,
    "existing_id": 100
  }
}
```

### 优点总结

- ✓ RESTful规范清晰
- ✓ 统一响应格式
- ✓ 错误处理完善
- ✓ 19个API端点覆盖完整

### 改进建议

1. 简化微信OAuth2流程（3种场景 → 2步流程）
2. 完善API版本控制策略（升级触发条件、废弃流程）
3. 修正文件去重响应码（409 → 200）

## 第4章：认证授权设计

### 发现的问题

#### CRITICAL - 5项安全增强措施过度设计

**问题描述：**
Round 1-3共识引入5项微信绑定安全措施：
1. 学生身份验证（短信/邮件/学生证）
2. 受限Token（scope: password_setup_only）
3. 事务锁（select_for_update）
4. 审计日志
5. 通用错误消息

**问题：**
- **措施1（身份验证）**：需要短信网关/邮件服务，增加依赖和成本
- **措施2（受限Token）**：增加Token类型，客户端需要特殊处理
- **措施3（事务锁）**：单实例部署下，竞态条件极罕见
- **措施4（审计日志）**：已有通用审计日志，无需特殊处理
- **措施5（通用错误）**：合理，但防止学号枚举的价值存疑（学号本就公开）

**影响范围：**
- 实施复杂度大幅增加
- 依赖外部服务（短信/邮件）
- 用户体验差（多步验证）
- 实际安全收益有限

**建议方案：**
**简化为2项核心措施**：
1. **密码验证**：微信绑定已有账户时需要密码验证（已有设计）
2. **审计日志**：记录所有绑定操作（已有通用机制）

**删除过度措施**：
- ❌ 学生身份验证：Phase 1不实施，依赖管理员导入用户
- ❌ 受限Token：统一使用标准JWT Token
- ❌ 事务锁：单实例场景下不需要
- ❌ 通用错误：返回具体错误，便于调试

**理由：**
- 本系统是**内部系统**，用户由管理员导入，不是公开注册
- 学号枚举风险低（学号本就在校内公开）
- 过度安全措施增加复杂度，收益有限

#### MAJOR - JWT Token 7天有效期过长

**问题描述：**
Access Token有效期7天，如果Token泄露，攻击者有7天窗口期。

**影响范围：**
- 安全风险高
- Token泄露后难以及时撤销

**建议方案：**
1. **缩短Access Token有效期**：1小时
2. **Refresh Token有效期**：7天
3. **自动刷新机制**：客户端在Token过期前自动刷新

```
Access Token: 1小时（短期，高频使用）
Refresh Token: 7天（长期，低频使用）
```

#### MAJOR - RBAC权限模型过于简单

**问题描述：**
权限矩阵只有3个角色（student/counselor/admin），但：
1. **辅导员权限范围不明确**：能否查看其他年级申请？
2. **学工部权限过大**：admin既能审批又能管理系统配置
3. **缺少权限细分**：无法实现"只读管理员"、"数据分析员"等角色

**建议方案：**
**细化权限模型**：
```python
# 角色定义
student          # 学生
counselor        # 辅导员（只能查看本年级）
counselor_lead   # 年级组长（可查看本学院所有年级）
admin_approver   # 学工部审批员（只能审批）
admin_manager    # 学工部管理员（审批+配置管理）
admin_analyst    # 数据分析员（只读所有数据）
```

**权限范围**：
```python
# 辅导员权限范围
class CounselorPermission:
    def has_permission(self, request, view):
        # 只能查看本年级学生申请
        return Application.objects.filter(
            student__grade=request.user.grade,
            student__department=request.user.department
        )
```

#### MINOR - 账户锁定30分钟可能过短

**问题描述：**
密码错误5次锁定30分钟，但：
- 30分钟后自动解锁，攻击者可继续尝试
- 缺少永久锁定机制

**建议方案：**
**渐进式锁定**：
- 第1次锁定：30分钟
- 第2次锁定：2小时
- 第3次锁定：24小时
- 第4次锁定：永久锁定（需管理员解锁）

### 优点总结

- ✓ 双通道认证设计合理
- ✓ JWT Token机制标准
- ✓ 密码加密使用bcrypt

### 改进建议

1. **简化微信绑定安全措施**：删除过度设计，保留核心2项
2. **缩短Access Token有效期**：7天 → 1小时
3. **细化RBAC权限模型**：3角色 → 6角色
4. **渐进式账户锁定**：固定30分钟 → 渐进式锁定
