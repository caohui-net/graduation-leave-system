**文档版本：** v1.0

## 一、对接概述

### 1.1 对接目的

本系统需要与学校现有的**宿舍管理系统**进行数据对接，实现以下功能：

- 学生提交离校申请时，自动验证宿舍是否已清退
- 验证通过后，允许学生继续提交申请
- 验证失败时，提示学生先完成宿舍清退手续

### 1.2 对接方式

本系统支持两种对接方式，按优先级排序：

**方式1：API接口对接（推荐）**
- 宿舍管理系统提供REST API接口
- 本系统调用API实时查询宿舍清退状态
- 优点：实时性高、自动化程度高
- 缺点：需要宿舍管理系统支持API开发

**方式2：数据文件导入（备选）**
- 宿舍管理系统定期导出清退数据文件
- 本系统导入数据文件到本地数据库
- 优点：实施简单、无需开发API
- 缺点：数据有延迟、需要人工操作

### 1.3 对接时间表

| 阶段 | 时间 | 任务 |
|------|------|------|
| 需求确认 | 第1周 | 确认对接方式、数据字段、接口规范 |
| 接口开发 | 第2-3周 | 宿舍管理系统开发API接口（如选择方式1） |
| 联调测试 | 第4周 | 双方系统联调测试 |
| 上线部署 | 第5周 | 正式上线 |

## 二、方式1：API接口对接（推荐）

### 2.1 接口规范

**接口名称：** 查询学生宿舍清退状态

**请求方式：** GET 或 POST

**请求地址：** `http://宿舍管理系统域名/api/dorm/checkout/status`

**认证方式：** 
- API Key认证（推荐）
- 或 Basic Auth认证
- 或 IP白名单

### 2.2 请求参数

| 参数名 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| student_id | String | 是 | 学号 | "2020010101" |
| name | String | 否 | 姓名（用于验证） | "张三" |
| id_card | String | 否 | 身份证号（用于验证） | "110101199001011234" |

**请求示例（GET）：**
```
GET /api/dorm/checkout/status?student_id=2020010101&name=张三
Host: dorm.example.com
Authorization: Bearer YOUR_API_KEY
```

**请求示例（POST JSON）：**
```json
POST /api/dorm/checkout/status
Host: dorm.example.com
Content-Type: application/json
Authorization: Bearer YOUR_API_KEY

{
  "student_id": "2020010101",
  "name": "张三"
}
```

### 2.3 响应参数

**成功响应（HTTP 200）：**

| 参数名 | 类型 | 说明 | 示例 |
|--------|------|------|------|
| code | Integer | 状态码（0=成功） | 0 |
| message | String | 响应消息 | "查询成功" |
| data | Object | 数据对象 | 见下表 |

**data对象字段：**

| 参数名 | 类型 | 说明 | 示例 |
|--------|------|------|------|
| student_id | String | 学号 | "2020010101" |
| name | String | 姓名 | "张三" |
| dorm_building | String | 宿舍楼栋 | "1号楼" |
| dorm_room | String | 宿舍房间号 | "101" |
| is_checked_out | Boolean | 是否已清退 | true |
| checkout_date | String | 清退日期（ISO 8601格式） | "2026-05-20T10:30:00Z" |
| checkout_status | String | 清退状态 | "completed" / "pending" / "not_started" |
| remarks | String | 备注 | "已完成清退，物品已搬离" |

**响应示例（已清退）：**
```json
{
  "code": 0,
  "message": "查询成功",
  "data": {
    "student_id": "2020010101",
    "name": "张三",
    "dorm_building": "1号楼",
    "dorm_room": "101",
    "is_checked_out": true,
    "checkout_date": "2026-05-20T10:30:00Z",
    "checkout_status": "completed",
    "remarks": "已完成清退，物品已搬离"
  }
}
```

**响应示例（未清退）：**
```json
{
  "code": 0,
  "message": "查询成功",
  "data": {
    "student_id": "2020010101",
    "name": "张三",
    "dorm_building": "1号楼",
    "dorm_room": "101",
    "is_checked_out": false,
    "checkout_date": null,
    "checkout_status": "not_started",
    "remarks": "尚未办理清退手续"
  }
}
```

**错误响应（HTTP 200，业务错误）：**
```json
{
  "code": 1001,
  "message": "学号不存在",
  "data": null
}
```

**错误响应（HTTP 4xx/5xx）：**
```json
{
  "code": 500,
  "message": "系统内部错误",
  "data": null
}
```

### 2.4 错误码说明

| 错误码 | 说明 | 处理方式 |
|--------|------|----------|
| 0 | 成功 | 正常处理 |
| 1001 | 学号不存在 | 提示学生检查学号 |
| 1002 | 参数错误 | 检查请求参数 |
| 1003 | 认证失败 | 检查API Key |
| 5000 | 系统内部错误 | 重试或联系管理员 |

### 2.5 接口性能要求

- **响应时间：** < 500ms（P95）
- **可用性：** > 99%
- **并发支持：** > 50 QPS
- **超时时间：** 5秒（本系统设置）

### 2.6 接口测试

**测试环境：**
- 测试地址：`http://dorm-test.example.com/api/dorm/checkout/status`
- 测试API Key：`test_api_key_123456`
- 测试学号：`2020010101`（已清退）、`2020010102`（未清退）

**测试工具：**

## 三、方式2：数据文件导入（备选）

### 3.1 文件格式

**文件类型：** Excel (.xlsx) 或 CSV (.csv)

**文件命名：** `dorm_checkout_YYYYMMDD.xlsx`（例如：`dorm_checkout_20260527.xlsx`）

**字符编码：** UTF-8（CSV文件）

**导出频率：** 每日一次（建议凌晨1点）

### 3.2 数据字段

| 字段名（中文） | 字段名（英文） | 数据类型 | 必填 | 说明 | 示例 |
|---------------|---------------|----------|------|------|------|
| 学号 | student_id | 文本 | 是 | 学生学号 | 2020010101 |
| 姓名 | name | 文本 | 是 | 学生姓名 | 张三 |
| 宿舍楼栋 | dorm_building | 文本 | 否 | 宿舍楼栋名称 | 1号楼 |
| 宿舍房间号 | dorm_room | 文本 | 否 | 宿舍房间号 | 101 |
| 是否已清退 | is_checked_out | 文本 | 是 | 是/否 | 是 |
| 清退日期 | checkout_date | 日期 | 否 | YYYY-MM-DD格式 | 2026-05-20 |
| 清退状态 | checkout_status | 文本 | 否 | 已完成/进行中/未开始 | 已完成 |
| 备注 | remarks | 文本 | 否 | 备注信息 | 已完成清退 |

### 3.3 Excel模板

**下载模板：** `docs/templates/宿舍清退数据模板.xlsx`

**模板示例：**

| 学号 | 姓名 | 宿舍楼栋 | 宿舍房间号 | 是否已清退 | 清退日期 | 清退状态 | 备注 |
|------|------|----------|------------|------------|----------|----------|------|
| 2020010101 | 张三 | 1号楼 | 101 | 是 | 2026-05-20 | 已完成 | 已完成清退，物品已搬离 |
| 2020010102 | 李四 | 1号楼 | 102 | 否 |  | 未开始 | 尚未办理清退手续 |
| 2020010103 | 王五 | 2号楼 | 201 | 是 | 2026-05-21 | 已完成 | 已完成清退 |

### 3.4 CSV格式示例

```csv
学号,姓名,宿舍楼栋,宿舍房间号,是否已清退,清退日期,清退状态,备注
2020010101,张三,1号楼,101,是,2026-05-20,已完成,已完成清退，物品已搬离
2020010102,李四,1号楼,102,否,,未开始,尚未办理清退手续
2020010103,王五,2号楼,201,是,2026-05-21,已完成,已完成清退
```

### 3.5 数据要求

**数据完整性：**
- 必填字段不能为空
- 学号必须唯一
- 日期格式必须正确

**数据准确性：**
- 学号必须与学生信息系统一致
- 姓名必须与学生信息系统一致
- 清退状态必须真实准确

**数据时效性：**
- 建议每日更新一次
- 数据延迟不超过24小时

### 3.6 文件传输方式

**方式A：FTP/SFTP上传**
- 宿舍管理系统自动上传到指定FTP服务器
- 本系统定时下载并导入

**方式B：共享文件夹**
- 宿舍管理系统导出到共享文件夹
- 本系统定时读取并导入

**方式C：手动上传**
- 管理员手动下载文件
- 在本系统管理后台上传导入

## 四、数据安全要求

### 4.1 传输安全

- API接口必须使用HTTPS协议
- 文件传输必须使用SFTP或加密传输
- API Key必须妥善保管，不得泄露

### 4.2 访问控制

- API接口限制IP白名单访问
- 文件传输账号独立，权限最小化
- 定期更换API Key和传输密码

### 4.3 数据脱敏

- 导出数据不包含敏感信息（身份证号、手机号等）
- 仅包含必要的业务数据

### 4.4 审计日志

- 记录所有API调用日志
- 记录所有文件导入日志
- 日志保留至少90天

## 六、上线部署

### 6.1 生产环境配置

**API接口方式：**
- 生产环境API地址：`https://dorm.example.com/api/dorm/checkout/status`
- 生产环境API Key：由宿舍管理系统提供
- IP白名单：本系统服务器IP

**文件导入方式：**
- FTP/SFTP地址：由宿舍管理系统提供
- 传输账号密码：由宿舍管理系统提供
- 文件存放路径：`/data/dorm_checkout/`

### 6.2 监控告警

**监控指标：**
- API调用成功率
- API响应时间
- 文件导入成功率
- 数据更新时效性

**告警规则：**
- API调用失败率 > 5%，发送告警
- API响应时间 > 1秒，发送告警
- 文件导入失败，发送告警
- 数据超过48小时未更新，发送告警

### 6.3 应急预案

**API不可用：**
- 自动切换到降级模式（手动上传证明）
- 通知管理员和宿舍管理系统负责人
- 记录降级日志

**数据异常：**
- 暂停自动验证
- 人工审核宿舍清退证明
- 排查数据问题
