### 问题3：实现范围是否过大？

**当前计划包含：**
1. 签名客户端（核心）
2. 诊断脚本
3. Mock测试
4. 错误处理

**质疑：** 是否应该更窄？

**选项A：** 按原计划全部实现
- 优点：一次性完成，架构完整
- 缺点：工作量大，调试周期长

**选项B：** 分步实现
  - B1: 先只实现签名函数 + 官方样例验证
  - B2: 再实现请求构造 + 响应解析
  - B3: 最后补充错误处理 + Mock测试
- 优点：每步验收清晰，风险可控
- 缺点：需要多次讨论

**选项C：** 最小MVP
  - 只实现签名验证脚本
  - 手动构造curl命令测试真实API
  - 验证通过后再考虑封装客户端
- 优点：最快验证可行性
- 缺点：不够工程化，后续还需重构

**Claude倾向：** 选项A（原计划），因为已有明确需求
