## 一、三方观点整合

### 1.1 用户策略（原始）

**核心主张：**
> "文件1与文件2有共同字段（学生姓名+所在学院），应以文件1为基准，将文件2中的学号按共有字段内容进行比对添加，得到新学生文件（文件5）。"

**关联关系：**
- File5（学生） ← File1 + File2（姓名+学院关联）
- File5楼栋 → File3楼栋号 → 宿管员
- File5学院 → File4学院 → 辅导员

**预期结果：** "如此应该满足项目需求"

### 1.2 Claude初始分析（已废弃）

**错误结论：**

- ❌ "存在3个P0缺陷，数据源部分满足"
- ❌ "推荐方案C（联系数据提供方）85分"

**纠正说明：**
Claude未发现File2的FY/XSH字段包含学院信息，只检查了YXMC（校名），导致错误判断"两文件无共同字段"。

### 1.3 Codex审查结论（权威）

**核心判断：**
- ✓ 用户方向正确，File1/File2确实可关联
- ⚠️ 原始"姓名+学院"不足，必须升级为"姓名+规范化学院+班级/BH"

- ⚠️ 剩余271行研究生无法自动匹配（File1班级为空，File2无研究生）
- ⚠️ 楼栋→宿管员一对多，不应塞入单一班级字段
- ⚠️ 学院→辅导员映射可生成，但属业务降级，需确认

**数据质量评级：** B-

**推荐方案：** 用户策略增强版 > 重新获取数据

## 二、最终技术方案

### 2.1 数据关联规则（生产级）

**安全匹配键：**
```python
match_key = (
    student_name,
    normalize_college(college_name),  # 规范化处理
    class_id_or_bh  # 必须非空
)
```

**学院规范化表（19个学院）：**
| File1学院名称 | 规范化后 | File2/File4对应 |
|--------------|---------|----------------|
| 文学院（苏东坡书院） | 文学院(苏东坡书院) | FY/学院 |
| 政法学院、纪检监察学院、知识产权学院 | 政法学院 | FY/学院 |
| 计算机学院 | 计算机学院 | 计算机与人工智能学院 |
| ... | ... | ... |

**匹配结果：**
- 自动匹配：5,559行（95.35%）
- 研究生待补：271行（4.65%，File1班级为空）
- File2独有：116行（需确认是否住校）

### 2.2 审批路由策略

**辅导员路由（学院级）：**
```
class_id -> 规范化学院 -> File4辅导员工号
```
- 可为196个非空班级生成映射
- **业务降级**：同学院所有班级由1名辅导员统一审批
- **前置条件**：业务方确认接受学院统一审批

**宿管员路由（楼栋级）：**
```
student.building -> File3楼栋号 -> List[宿管员] -> 选主规则
```
- 覆盖33/33楼栋
- **问题**：每栋多个宿管员（一对多）
- **解决方案**：
  1. 确认选主规则（主责/轮值/任选其一）
  2. 或支持多审批人
  3. 或在申请时动态路由（不预存class_id映射）

**研究生路由（特殊处理）：**
```
graduate_students.class_id = "GRADUATE_2026"
GRADUATE_2026 -> File4研究生辅导员
building -> dorm_manager（同本科生）
```

### 2.3 数据完整性评估

| 数据表 | 覆盖率 | 准确率 | 质量等级 | 说明 |
|--------|--------|--------|----------|------|
| 学生表（本科） | 95.35% | ~100% | A | 需抽样验证 |
| 学生表（研究生） | 0% | N/A | F | 271行缺学号和class_id |
| 辅导员映射 | 100% | ⚠️ | B | 学院级≠班级级 |
| 宿管员映射 | 100% | ⚠️ | B | 一对多+跨楼栋问题 |
| **综合评级** | **95.35%** | **~100%** | **B-** | 本科可用，研究生待补 |

## 四、风险与缓解

### 风险1: 研究生无法使用系统

**风险等级：** P1  
**影响：** 271名研究生（4.65%）无学号和class_id

**缓解措施：**
- 方案A：研究生不纳入系统（确认后排除）
- 方案B：创建虚拟class_id="GRADUATE_2026"，学号人工补充（0.5-1天）
- 方案C：联系教务处获取研究生数据（时间不确定）

**推荐：** 方案B

### 风险2: 辅导员审批粒度不符

**风险等级：** P0  
**影响：** 学院统一审批 vs 班级负责制语义不一致

**缓解措施：**
- 方案A：业务确认接受学院统一审批（降级但可用）
- 方案B：向学院补充"班级→辅导员"真实映射表（需协调）

**推荐：** 优先方案A，如不接受则方案B

### 风险3: 宿管员路由与模型不匹配

**风险等级：** P0  
**影响：** 当前ClassMapping按class_id，但184/196班级跨多个楼栋

**缓解措施：**
- 方案A：改模型，分离楼栋路由（BuildingMapping）
- 方案B：申请时动态按学生楼栋路由，不预存映射
- 方案C：业务确认班级只对应主楼栋的主宿管员（降级）

**推荐：** 方案A或B，避免方案C
