## 发现的问题

### 问题1：class_name字段必填性不一致 [P2]

**位置：** `docs/数据对接说明文档.md:105`

**问题描述：**
当宿舍系统不一定提供班级信息时，文件导入字段表已把 `class_name` 标为"否"（非必填），但 API 响应字段表只给 `bed_number` 标注了"可选"，会让接口实现方把 `class_name` 误认为必返字段。

**影响：**
接口实现方可能将class_name作为必填字段处理，导致宿舍系统无法提供班级信息时接口调用失败。

**修复建议：**
- 方案A：在API响应字段表中也将class_name标注为"可选"
- 方案B：在所有文档中统一改为必填（需确认宿舍系统一定能提供）

---

### 问题2：Excel示例表格列错位 [P2]

**位置：** `docs/数据对接说明文档.md:241`

**问题描述：**
在"李四"这一行，表头只有 10 列，但当前内容在 `否` 后面放了两个空单元格，导致该行变成 11 列，`未开始` 和备注列会错位；这也与下面的 CSV 示例不一致。

**影响：**
- 用户按照Excel示例填写数据时会产生列对齐错误
- 导入系统时可能解析失败或数据错位

**修复建议：**
删除"李四"行中多余的空单元格，确保所有行都是10列。

---

### 问题3：系统设计文档字段命名不一致 [P2]

**位置：** `docs/design/2026-05-27-system-design.md:1636-1639`

**问题描述：**
系统设计文档中的响应仍使用 `is_cleared`、`clearance_date`、`room_no`，而数据对接文档定义的是 `is_checked_out`、`checkout_date`、`dorm_building`、`dorm_room` 等字段；在补充 `class_name` 和 `bed_number` 后仍保留两套命名，会让实现方按不同文档接出不兼容的 payload。

**影响：**
- 前后端开发人员按照不同文档实现，导致接口字段不匹配
- 集成测试时发现字段名错误，需要返工修改

**修复建议：**
统一两个文档的字段命名，建议采用数据对接文档的命名（is_checked_out、checkout_date等），因为这是对外接口规范。
