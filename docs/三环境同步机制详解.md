## 概述

### 三环境架构

本项目采用三环境部署架构：

| 环境 | 服务器 | 用途 | 端口 |
|------|--------|------|------|
| **开发环境** | 172.17.12.199 | 代码开发、本地测试 | - |
| **Staging环境** | 172.17.12.196 | 预生产测试、验证 | 17787/17788/15432 |
| **Production环境** | 172.17.12.196 | 正式对外服务 | 7787/7788/5432 |

### 核心特点

- ✅ **环境隔离**: 不同端口、独立数据库、独立配置
- ✅ **自动同步**: 开发→Staging自动同步（<10秒）
- ✅ **安全促销**: Staging→Production手动触发，带备份回滚
- ✅ **独立部署**: 不依赖Git push/pull，基于文件系统同步

## 同步机制

### 机制1: 开发→Staging (自动同步)

#### 触发条件

文件变更事件（修改、创建、删除、移动）

#### 工作流程

```
1. inotify监听文件系统事件
   ↓
2. 1秒防抖（避免频繁触发）
   ↓
3. 执行sync-to-staging.sh
   ↓
4. rsync同步到196:/opt/graduation-leave-system/staging/
   ↓
5. 记录日志到/tmp/sync-to-staging.log
```

#### 核心命令

```bash
# 监听脚本
inotifywait -m -r -e modify,create,delete,move \
  --exclude '\.git|__pycache__|\.pyc$|venv|node_modules|\.claude|\.omc|\.wolf|\.trellis|\.collab' \
  /home/caohui/projects/graduation-leave-system

# 同步命令
rsync -avz --delete \
  --exclude='.git' \
  --exclude='backend/venv' \
  --exclude='backend/__pycache__' \
  --exclude='backend/*.pyc' \
  --exclude='.env.dev' \
  --exclude='.env.prod' \
  --exclude='node_modules' \
  --exclude='.claude' \
  --exclude='.omc' \
  --exclude='.wolf' \
  --exclude='.trellis' \
  --exclude='.collab' \
  /home/caohui/projects/graduation-leave-system/ \
  caohui@172.17.12.196:/opt/graduation-leave-system/staging/
```

#### 同步内容

**包含**:
- ✅ 所有源代码文件 (.py, .js, .vue, .ts)
- ✅ Migration文件 (backend/apps/*/migrations/*.py)
- ✅ 配置示例 (.env.example, .env.docker)
- ✅ 静态资源 (CSS, 图片, 字体)
- ✅ 文档文件 (docs/, README.md)

**排除**:
- ❌ .git目录 - Git历史不同步
- ❌ venv/node_modules - 依赖包由容器安装
- ❌ __pycache__/*.pyc - Python缓存文件
- ❌ .env.dev/.env.prod - 环境特定配置
- ❌ .claude/.omc/.wolf - 开发工具配置
- ❌ logs/backups - 日志和备份

#### 性能特点

- **延迟**: <10秒（从保存到同步完成）
- **防抖**: 1秒内多次修改只触发一次同步
- **带宽**: 增量同步，只传输变更部分
- **可靠性**: rsync保证数据一致性

#### 监控与日志

```bash
# 查看同步服务状态
systemctl --user status sync-to-staging.service

# 查看实时日志
tail -f /tmp/sync-to-staging.log

# 最近同步记录
tail -20 /tmp/sync-to-staging.log
```

### 机制2: Staging→Production (手动促销)

#### 触发条件

手动执行promote脚本（验证通过后）

#### 工作流程

```
1. 健康检查Staging环境
   ↓
2. 备份Production当前状态到 /opt/backups/YYYYMMDD_HHMMSS/
   ↓
3. rsync同步Staging→Production
   ↓
4. 重启Production容器
   ↓
5. 健康检查Production
   ↓
6. 成功 → 完成 | 失败 → 自动回滚
```

#### 执行命令

```bash
# 从开发机远程执行
ssh caohui@172.17.12.196 "/opt/graduation-leave-system/scripts/promote-to-prod.sh"

# 或登录196后本地执行
ssh caohui@172.17.12.196
/opt/graduation-leave-system/scripts/promote-to-prod.sh
```

#### 安全机制

**1. 备份**:
```bash
# 自动备份位置
/opt/graduation-leave-system/backups/YYYYMMDD_HHMMSS/production/

# 备份内容
- 完整代码目录
- docker-compose.yml
- .env.docker配置
```

**2. 健康检查**:
```bash
# Staging检查（促销前）
curl http://localhost:17787/readyz
→ 必须返回 {"status": "ready", "database": "connected"}

# Production检查（促销后）
curl http://localhost:7787/readyz
→ 失败则触发回滚
```

**3. 自动回滚**:
- 健康检查失败时自动触发
- 从备份目录恢复代码
- 重启容器到旧版本
- 记录回滚日志

#### 同步内容

与开发→Staging相同，完整同步Staging目录到Production

## 数据库迁移

### 重要概念

**Migration文件同步 ≠ 数据库变更**

```
代码同步流程:
models.py修改 → makemigrations → 生成0008.py → rsync同步到各环境
                                                      ↓
                                     文件存在，但数据库未变更

数据库迁移流程:
各环境独立执行: docker compose exec backend python manage.py migrate
                                      ↓
                            数据库Schema变更生效
```

### Django Migration两步骤

#### 步骤1: makemigrations (生成迁移文件)

**在开发环境执行**:
```bash
cd /home/caohui/projects/graduation-leave-system/backend
python3 manage.py makemigrations

# 输出示例
# Migrations for 'applications':
#   apps/applications/migrations/0008_application_application_type_and_more.py
#     - Add field application_type to application
#     - Add field stay_end_date to application
#     - Add field stay_reason to application
```

**作用**:
- 检测models.py变更
- 生成迁移文件 (.py)
- 文件提交到Git（版本控制）
- **不改变数据库**

#### 步骤2: migrate (应用迁移)

**在每个环境独立执行**:

**开发环境**:
```bash
cd /home/caohui/projects/graduation-leave-system/backend
python3 manage.py migrate applications
```

**Staging环境**:
```bash
ssh caohui@172.17.12.196
cd /opt/graduation-leave-system/staging
docker compose exec backend python manage.py migrate applications
```

**Production环境**:
```bash
ssh caohui@172.17.12.196
cd /opt/graduation-leave-system/production
docker compose exec backend python manage.py migrate applications
```

**作用**:
- 读取迁移文件
- 执行SQL变更
- 更新django_migrations表
- **实际改变数据库Schema**

### Migration文件传播流程

```
1. 开发环境(199)
   - 修改models.py
   - 执行makemigrations
   - 生成0008.py文件
   ↓ (rsync自动同步 <10秒)
   
2. Staging环境(196)
   - 接收0008.py文件
   - 文件存在，数据库未变更
   - 需手动执行migrate
   ↓ (promote脚本手动触发)
   
3. Production环境(196)
   - 接收0008.py文件
   - 文件存在，数据库未变更
   - 需手动执行migrate
```

### 验证Migration状态

```bash
# 查看所有migrations状态
python manage.py showmigrations

# 查看特定app
python manage.py showmigrations applications

# 输出示例
# applications
#  [X] 0001_initial
#  [X] 0002_allow_null_class_id
#  [X] 0007_merge_20260620_0217
#  [X] 0008_application_application_type_and_more  ← [X]表示已应用
#  [ ] 0009_new_migration                           ← [ ]表示未应用
```

### 三环境数据库独立性

| 环境 | 数据库地址 | 数据卷 | 数据独立性 |
|------|-----------|--------|----------|
| 开发 | localhost:5432 | local_postgres_data | ✅ 独立 |
| Staging | 172.17.12.196:15432 | staging_postgres_data | ✅ 独立 |
| Production | 172.17.12.196:5432 | prod_postgres_data | ✅ 独立 |

**关键点**:
- 三个数据库完全独立
- Migration需要在每个环境单独执行
- 数据不会自动同步
- 测试数据不会进入生产库

## 故障处理

### 问题1: 同步服务未运行

**症状**: 代码修改后Staging未更新

**解决**:
```bash
systemctl --user restart sync-to-staging.service
~/scripts/sync-to-staging.sh  # 手动触发
```

### 问题2: Staging服务无响应

**症状**: 健康检查失败

**解决**:
```bash
ssh caohui@172.17.12.196
cd /opt/graduation-leave-system/staging
docker compose restart
sleep 10
curl http://localhost:17787/readyz
```

### 问题3: 容器内代码未更新

**原因**: 文件同步了，但容器未重启

**解决**:
```bash
cd /opt/graduation-leave-system/staging
docker compose restart backend
```

### 问题4: Migration冲突

**解决**:
```bash
# 开发环境合并
python3 manage.py makemigrations --merge

# 或回滚后重新应用
docker compose exec backend python manage.py migrate applications 0007
docker compose exec backend python manage.py migrate
```

## 最佳实践

### 开发流程

1. **修改代码** → 自动同步到Staging (<10秒)
2. **Staging测试** → 验证功能和兼容性
3. **手动promote** → 同步到Production
4. **Production验证** → 健康检查通过

### Migration管理

1. **开发环境**:
   - 修改models.py
   - 执行makemigrations
   - 执行migrate（本地验证）
   - Git commit migration文件

2. **Staging环境**:
   - Migration文件自动同步
   - 手动执行migrate
   - 测试数据库变更

3. **Production环境**:
   - Migration文件通过promote同步
   - 备份数据库
   - 手动执行migrate
   - 验证功能

### 安全建议

1. **Production操作前**:
   - ✅ 先在Staging完整测试
   - ✅ 备份数据库
   - ✅ 记录当前版本

2. **Migration操作**:
   - ✅ 先查看showmigrations
   - ✅ 了解变更内容
   - ✅ 准备回滚方案

3. **促销操作**:
   - ✅ 确认Staging健康
   - ✅ 选择低峰期执行
   - ✅ 准备快速回滚

### 监控要点

- 开发→Staging同步延迟 (<10秒正常)
- Staging/Production健康检查
- 容器运行时间 vs 代码更新时间
- Migration应用状态一致性
