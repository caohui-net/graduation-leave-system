## 2. 监控告警缺失

### 2.1 原方案监控空白

**当前**: 无监控，只有事后日志

```bash
# 唯一的"监控"是查日志
tail -f /tmp/sync-to-prod.log
journalctl --user -u sync-to-prod.service -f
```

**问题**:
- 同步失败不会主动告警
- 服务异常需人工发现
- 无性能指标（CPU/内存/磁盘）
- 无业务指标（请求量/错误率）

### 2.2 必需的监控指标

```yaml
# 建议监控指标清单
infrastructure:
  - disk_usage: alert_threshold: 80%
  - memory_usage: alert_threshold: 85%
  - inotify_process: check_interval: 60s
  - rsync_success_rate: alert_threshold: 95%
  
application:
  - backend_health: /readyz endpoint, timeout: 5s
  - frontend_response: port 7788, timeout: 2s
  - database_connections: max_conn check
  
sync:
  - last_sync_timestamp: stale_threshold: 300s
  - sync_failure_count: alert_threshold: 3
  - file_count_diff: 199 vs 196 delta > 10
```

### 2.3 最小告警方案（低成本）

```bash
# /home/caohui/scripts/health-check.sh
#!/bin/bash
# 每5分钟执行，检查关键指标

ALERT_WEBHOOK="https://your-webhook.com/alert"  # 企业微信/钉钉

# 1. 检查同步服务
if ! systemctl --user is-active sync-to-prod.service > /dev/null; then
    curl -X POST "$ALERT_WEBHOOK" -d '{"msg":"❌199同步服务已停止"}'
fi

# 2. 检查同步延迟
LAST_SYNC=$(stat -c %Y /tmp/sync-to-prod.log)
NOW=$(date +%s)
if [ $((NOW - LAST_SYNC)) -gt 600 ]; then  # 10分钟无同步
    curl -X POST "$ALERT_WEBHOOK" -d '{"msg":"⚠️代码同步超时10分钟"}'
fi

# 3. 检查196服务
if ! ssh root@172.17.12.196 "curl -sf http://localhost:7787/readyz > /dev/null"; then
    curl -X POST "$ALERT_WEBHOOK" -d '{"msg":"🔴196后端服务异常"}'
fi

# 4. 检查磁盘空间
DISK_USAGE=$(df / | tail -1 | awk '{print $5}' | sed 's/%//')
if [ "$DISK_USAGE" -gt 85 ]; then
    curl -X POST "$ALERT_WEBHOOK" -d "{\"msg\":\"⚠️磁盘使用${DISK_USAGE}%\"}"
fi
```


## 3. 故障恢复流程缺陷

### 3.1 场景分析

#### 场景1: inotify服务崩溃

**当前方案**: 无自动重启（systemd Restart=on-failure 但有延迟）

**问题**: 崩溃期间代码变更丢失

**改进**:
```ini
# systemd服务增强
[Service]
Restart=always
RestartSec=5
StartLimitInterval=200
StartLimitBurst=5

# 健康检查（每30秒写心跳）
ExecStartPost=/bin/bash -c 'while true; do date +%s > /tmp/sync-watchdog; sleep 30; done'
```

```bash
# 外部看门狗
# /etc/cron.d/sync-watchdog
* * * * * caohui [ $(( $(date +%s) - $(cat /tmp/sync-watchdog) )) -gt 90 ] && systemctl --user restart sync-to-prod.service
```

#### 场景2: 网络分区（199与196断连）

**当前方案**: rsync失败，写日志，无重试

**问题**: 断网期间变更堆积，恢复后可能冲突

**改进**:
```bash
# 增强同步脚本（带重试+队列）
sync_with_retry() {
    local max_attempts=3
    local attempt=1
    
    while [ $attempt -le $max_attempts ]; do
        if rsync ... ; then
            echo "[OK] Synced at attempt $attempt"
            return 0
        fi
        echo "[RETRY] Attempt $attempt failed, waiting 30s..."
        sleep 30
        ((attempt++))
    done
    
    # 失败后记录到队列
    echo "$(date +%s)" >> /tmp/sync-failed-queue.txt
    return 1
}
```

#### 场景3: 196磁盘满导致rsync失败

**当前方案**: 无磁盘检查，rsync静默失败

**改进**:
```bash
# 同步前检查196磁盘空间
check_remote_disk() {
    REMOTE_DISK=$(ssh root@172.17.12.196 "df / | tail -1 | awk '{print \$5}' | sed 's/%//'")
    if [ "$REMOTE_DISK" -gt 85 ]; then
        echo "[ERROR] Remote disk ${REMOTE_DISK}%, aborting sync"
        # 触发告警 + 自动清理
        ssh root@172.17.12.196 "docker system prune -af --volumes"
        return 1
    fi
}
```

#### 场景4: 误删关键文件触发同步

**当前方案**: `rsync --delete` 会同步删除到196

**风险**: 199误删 → 196自动删除 → 生产故障

**改进**:
```bash
# rsync添加保护机制
rsync -avz --delete \
  --backup --backup-dir=/opt/backup/rsync-$(date +%Y%m%d) \  # 备份被删除的文件
  --exclude='*.sql' \  # 关键文件禁止删除
  ...
```

## 4. 配置漂移风险

### 4.1 问题根源

```
199开发环境              196生产环境
├── .env.dev            ├── .env.prod
├── docker-compose.yml  ├── docker-compose.prod.yml
└── 经常修改              └── 较少修改

↓ 时间推移 ↓
配置分化 → 行为不一致 → 生产故障难复现
```

### 4.2 配置管理最佳实践

```bash
# 方案: 基础配置统一，差异通过覆盖
graduation-leave-system/
├── docker-compose.base.yml      # 公共配置
├── docker-compose.dev.yml       # 开发差异
├── docker-compose.prod.yml      # 生产差异
└── .env.template                 # 配置模板（不含敏感值）

# 启动方式
docker-compose -f docker-compose.base.yml -f docker-compose.dev.yml up   # 199
docker-compose -f docker-compose.base.yml -f docker-compose.prod.yml up  # 196
```

**好处**:
- 公共配置单一来源
- 差异显式声明
- 易于对比和审查

## 5. 日志管理缺失

### 5.1 当前日志问题

| 日志类型 | 位置 | 轮转 | 保留期 | 问题 |
|---------|------|------|--------|------|
| 同步日志 | /tmp/sync-to-prod.log | ❌ 无 | 永久 | 磁盘堆积 |
| systemd日志 | journalctl | ✅ 自动 | 7天 | ✅ 正常 |
| 前端日志 | /var/log/graduation-frontend.log | ❌ 无 | 永久 | 磁盘堆积 |
| 后端日志 | Docker容器 | ⚠️ 默认json-file | 无限 | 磁盘风险 |

### 5.2 日志轮转配置

```bash
# /etc/logrotate.d/graduation-sync
/tmp/sync-to-prod.log {
    daily
    rotate 7
    compress
    missingok
    notifempty
    create 0640 caohui caohui
}

/var/log/graduation-frontend*.log {
    daily
    rotate 7
    compress
    delaycompress
    missingok
    notifempty
}
```

```yaml
# docker-compose.prod.yml
services:
  backend:
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"
```

## 6. 灰度发布缺失

**当前方案**: 199改代码 → 自动同步196 → 立即生效

**问题**: 无测试缓冲，一旦有bug直接影响生产

**建议架构**:
```
199开发    →    196测试环境(端口17787)    →    196生产环境(端口7787)
   ↓                     ↓                              ↓
  自动同步            手动验证通过                  手动切流量
```

**实现方式**:
```bash
# 196运行两套环境
/opt/graduation-leave-system/
├── current/          # 当前生产（端口7787）
├── staging/          # 测试环境（端口17787）
└── releases/         # 历史版本（回滚用）

# 同步到staging，验证后切换
rsync ... 199 → 196/staging/
ssh 196 "cd staging && docker-compose up -d"
curl http://196:17787/readyz  # 测试通过
ssh 196 "ln -sfn staging current && systemctl restart graduation-*"
```
